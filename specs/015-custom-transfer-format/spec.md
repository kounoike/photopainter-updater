# 機能仕様: 独自画像転送形式追加

**Feature Branch**: `015-custom-transfer-format`  
**Created**: 2026-03-29  
**Status**: Draft  
**Input**: ユーザー記述: "HTTPサーバとファームウエア両方のコードから考えて、HTTP転送するデータをBMPなどの汎用形式に捉われず独自形式とすることで、ファームウエアでSDカードへの保存を不要とする形式を考えて。HTTPサーバでは既存の/, /image.bmpエントリポイントは今まで通りのBMP形式での返却とし、/image.binでアクセスしたときだけ独自形式で転送するようにして"

**記述ルール**: この文書は日本語で記述する。固有名詞、コード識別子、ライブラリ名のみ原文維持可。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 保存なしで表示更新したい (Priority: P1)

利用者は firmware が HTTP 応答を受けながらそのまま表示更新処理へ渡せるようにし、更新のたびに SD カードへ中間ファイルを書かずに済ませたい。

**Why this priority**: 中間保存を前提にした汎用画像形式の受け渡しをやめられれば、更新時間短縮、書き込み負荷削減、SD カード依存の低減につながるため。

**Independent Test**: firmware が独自形式の取得先を使う状態で画像更新を実行し、SD カード上に中間画像ファイルを作らずに表示更新が完了することを確認する。

**Acceptance Scenarios**:

1. **Given** firmware が独自形式の取得先を使う状態, **When** 利用者が更新を実行する, **Then** デバイスは SD カード上の中間画像ファイル保存を必須とせずに表示更新を完了できる。
2. **Given** 独自形式を使う更新経路が有効な状態, **When** 更新処理が正常完了する, **Then** 利用者は従来と同等の最終表示結果を確認できる。

---

### User Story 2 - 既存の BMP クライアントを壊したくない (Priority: P2)

利用者または既存クライアントは、新しい転送形式が追加されても、既存の `/` と `/image.bmp` からは今までどおり BMP を取得したい。

**Why this priority**: firmware 向け最適化のために既存の HTTP 取得互換性を壊すと、現在の利用方法や検証手順が崩れるため。

**Independent Test**: 既存の `/` と `/image.bmp` へアクセスし、変更前と同じ種類の応答が返ることを確認した上で、`/image.bin` のみが独自形式を返すことを確認する。

**Acceptance Scenarios**:

1. **Given** HTTP サーバが新機能を持つ状態, **When** 利用者が `/` または `/image.bmp` にアクセスする, **Then** サーバは従来どおり BMP 応答を返す。
2. **Given** HTTP サーバが新機能を持つ状態, **When** firmware 向けに `/image.bin` へアクセスする, **Then** サーバは BMP ではない独自形式の応答を返す。

---

### User Story 3 - 失敗時に安全に切り分けたい (Priority: P3)

運用者と開発者は、独自形式の取得に失敗した場合でも、表示更新を壊したまま進めず、通信問題か入力問題か、形式不整合かを切り分けたい。

**Why this priority**: 独自形式は BMP より専用性が高いため、失敗時の保護と診断導線がないと firmware 側の更新経路が不安定になるため。

**Independent Test**: 独自形式の取得失敗、途中中断、不正内容を個別に発生させ、失敗時に更新が安全に停止し、切り分け可能な結果を確認できることを確認する。

**Acceptance Scenarios**:

1. **Given** 独自形式の応答が途中で途切れる、または不正な内容を含む状態, **When** firmware が更新を実行する, **Then** デバイスは不完全な内容で表示更新を成功扱いにしない。
2. **Given** 独自形式での更新が失敗した状態, **When** 利用者または開発者が結果を確認する, **Then** 失敗が通信起因、入力起因、形式起因のいずれかを判別できる。

### Edge Cases

- `/image.bin` の応答が空、途中切断、または必要情報不足でも、firmware は部分更新を成功扱いにしないこと。
- `image.png` が未配置または変換不能な場合、BMP 経路と独自形式経路のどちらでも入力起因の失敗として判別できること。
- 独自形式の追加によって `/` と `/image.bmp` の応答内容、取得先、運用手順が変わらないこと。
- firmware が独自形式を利用できない状態でも、既存の BMP 経路へ戻す判断や切り分けができること。
- 同じ入力画像から生成した独自形式の結果が、従来の BMP 経路と見た目上で食い違わないこと。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST HTTP サーバと firmware の両方の都合を踏まえ、SD カードへの中間保存を不要にできる独自形式の転送契約を定義しなければならない。
- **FR-002**: System MUST `/image.bin` へのアクセス時だけ、firmware が受信しながら表示更新処理へ渡せる独自形式を返さなければならない。
- **FR-003**: System MUST 独自形式の応答だけで、firmware が追加の汎用画像デコードや中間ファイル保存を必須とせずに更新完了できるようにしなければならない。
- **FR-004**: System MUST `/` と `/image.bmp` では従来どおり BMP 応答を返し続けなければならない。
- **FR-005**: System MUST 同じ入力画像に対して、独自形式経路と BMP 経路が利用者にとって同等の最終表示結果へ到達できるようにしなければならない。
- **FR-006**: System MUST 独自形式に、firmware が受信途中の妥当性確認と完了判定を行うための識別情報を含めなければならない。
- **FR-007**: System MUST 独自形式の取得失敗、途中中断、不正内容時に、安全に更新を中止し、成功扱いにしない振る舞いを提供しなければならない。
- **FR-008**: System MUST 運用者または開発者が、独自形式経路の失敗を通信、入力画像、形式不整合の観点で切り分けられるようにしなければならない。
- **FR-009**: Users MUST be able to firmware 側で独自形式経路を使うか、既存 BMP 経路を使うかを明確に判断できなければならない。
- **FR-010**: System MUST 今回の範囲では新しい画像編集 UI、複数画像管理、追加の汎用画像取得先変更を要求してはならない。

### Key Entities *(include if feature involves data)*

- **独自形式応答**: `/image.bin` から返る firmware 向けの専用転送データ。表示更新に必要な情報と妥当性確認に必要な識別情報を含む。
- **BMP 互換応答**: `/` と `/image.bmp` から返る既存互換の応答。従来クライアント向けの比較基準として扱う。
- **更新セッション結果**: firmware が 1 回の画像更新を完了または失敗として判断する結果。成功、通信失敗、入力失敗、形式失敗を含む。
- **転送契約**: HTTP サーバと firmware の間で、どの取得先が何を返し、どう完了判定するかを定めた取り決め。

## Scope Boundaries *(mandatory)*

### Allowed Scope

- HTTP サーバに独自形式の取得先を追加すること
- firmware が中間保存なしで受信データを扱える更新経路を定義すること
- 既存 BMP 経路との互換性維持確認
- 失敗判定、診断導線、利用手順の更新

### Forbidden Scope

- `/` や `/image.bmp` を独自形式へ置き換えること
- 画像変換アルゴリズムそのものの品質要件を今回の主題から外れて変更すること
- 複数画像選択 UI やアップロード管理の追加
- `xiaozhi-esp32/` 配下の直接変更
- 独自形式追加と無関係なネットワーク設定、電源、ボタン仕様の変更

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: firmware 利用者は独自形式経路で 1 回の更新を実行したとき、SD カード上の中間画像ファイル保存なしで最終表示を更新できる。
- **SC-002**: 既存利用者は `/` と `/image.bmp` へアクセスしたとき、変更前と同じ BMP 応答を継続して取得できる。
- **SC-003**: 運用者は `/image.bin` の失敗時に、1 回の確認で通信、入力画像、形式不整合のいずれが原因かを切り分けられる。
- **SC-004**: 同じ入力画像に対し、独自形式経路と BMP 経路の最終表示結果が利用者視点で一致すると確認できる。

## Assumptions

- 対象 firmware は現在 HTTP 取得後に SD カード上の BMP を介して表示更新しており、この中間保存をなくす価値がある。
- 独自形式は firmware 向け最適化のために導入し、既存の人手確認や汎用クライアント確認は BMP 経路を継続利用する。
- 独自形式の詳細は planning で詰めるが、少なくとも順次受信処理と完了判定ができる契約を持つ前提とする。
- 既存の入力画像は引き続き単一画像を前提とし、複数画像配信や追加 UI は今回対象外とする。

## Documentation Impact

- server 側の取得先一覧と `/image.bin` の用途を説明する文書更新が必要になる。
- firmware 側で独自形式経路を使う条件、失敗時の切り分け、既存 BMP 経路との使い分けを文書へ反映する必要がある。
- 独自形式の転送契約と完了判定条件を設計成果物へ記録する必要がある。
