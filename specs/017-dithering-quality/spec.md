# 機能仕様: ディザリングアルゴリズムの改善

**Feature Branch**: `017-dithering-quality`
**Created**: 2026-03-30
**Status**: Draft

## User Scenarios & Testing *(mandatory)*

### User Story 1 - B案（Atkinsonアルゴリズム）の評価 (Priority: P1)

現行の Floyd-Steinberg ディザリングを Atkinson アルゴリズムに切り替えて、ePaper ディスプレイ上の画質を実機で評価できる状態にする。

**Why this priority**: 実装コストが最小でありながら粒状感改善の効果が見込まれるため、最初に評価する。

**Independent Test**: B案を有効にしてサーバーを起動し、ePaper に画像を表示して現行と視覚的に比較する。

**Acceptance Scenarios**:

1. **Given** B案が有効な状態でサーバーが起動している, **When** ePaper に画像が表示される, **Then** 現行（Floyd-Steinberg）と異なるディザリングパターンが確認できる
2. **Given** B案が有効な状態, **When** 写真・グラフィック両方の画像を表示する, **Then** 表示が正常に完了し、崩れ・エラーが発生しない

---

### User Story 2 - A案（CIE Lab色空間）の評価 (Priority: P1)

色距離計算を RGB ユークリッド距離から CIE Lab 色空間へ変更し、視覚的により自然な色選択によって粒状感が改善されるか実機で評価できる状態にする。

**Why this priority**: B案と同じ優先度で独立評価が必要。色距離改善は誤差拡散の基盤に影響するため重要。

**Independent Test**: A案を有効にしてサーバーを起動し、ePaper に画像を表示して現行と視覚的に比較する。

**Acceptance Scenarios**:

1. **Given** A案が有効な状態でサーバーが起動している, **When** ePaper に画像が表示される, **Then** 現行と比較して色の選択がより自然に見える
2. **Given** A案が有効な状態, **When** 写真・グラフィック両方の画像を表示する, **Then** 表示が正常に完了し、崩れ・エラーが発生しない

---

### User Story 3 - A+B組み合わせの評価 (Priority: P2)

A案（CIE Lab）と B案（Atkinson）を同時に有効化した状態で実機評価できる状態にする。

**Why this priority**: A案・B案それぞれの単独評価後に、組み合わせ効果を確認するため。

**Independent Test**: A+B を有効にしてサーバーを起動し、A案のみ・B案のみ・現行の3パターンと視覚的に比較する。

**Acceptance Scenarios**:

1. **Given** A案と B案の両方が有効な状態, **When** ePaper に画像が表示される, **Then** 表示が正常に完了し、単独案との違いが確認できる

---

### User Story 4 - 評価結果に基づくコードのクリーンアップ (Priority: P3)

評価で決定したアルゴリズムを正式採用し、評価用の切り替えコードや不要な実装を除去して、コードベースをクリーンな状態にする。

**Why this priority**: 評価完了後に実施する。最終的な保守性のために必要。

**Independent Test**: クリーンアップ後のコードで画像が正常に表示され、評価用の切り替え機構が残っていないことを確認する。

**Acceptance Scenarios**:

1. **Given** 採用アルゴリズムが決定している, **When** コードをクリーンアップする, **Then** 切り替え機構が除去され、採用アルゴリズムのみが残る
2. **Given** クリーンアップ完了後, **When** サーバーを起動して画像を表示する, **Then** 表示品質がクリーンアップ前と同等である

---

### Edge Cases

- 入力画像が極端に暗い・明るい・モノクロの場合でもディザリングが正常に動作すること
- パレット外の色を含む画像でも最近傍色選択が正しく機能すること
- アルゴリズム切り替えによってサーバー処理時間が著しく増加しないこと（ユーザー体験に影響しない範囲）

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST B案（Atkinsonアルゴリズム）を単独で有効化できる
- **FR-002**: System MUST A案（CIE Lab色空間での色距離計算）を単独で有効化できる
- **FR-003**: System MUST A案と B案を同時に有効化できる
- **FR-004**: System MUST 各アルゴリズムで生成された画像がバイナリフレーム形式で正しく出力される
- **FR-005**: System MUST 評価完了後、採用アルゴリズム以外の実装コードを除去できる
- **FR-006**: System MUST 現行（Floyd-Steinberg + RGB距離）を評価期間中のデフォルトとして維持できる

## Scope Boundaries *(mandatory)*

### Allowed Scope

- サーバー側のディザリングアルゴリズム変更（誤差拡散処理）
- 色距離計算の色空間変更（RGB → CIE Lab）
- 評価用のアルゴリズム切り替え機構の追加
- 評価完了後の不要コード除去

### Forbidden Scope

- ファームウェアの変更
- バイナリフレームフォーマット（PPBF）の変更
- 6色パレット自体の変更
- 彩度ブースト処理の変更
- サーバーの HTTP ルーティングやエンドポイントの変更

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: B案・A案・A+B の各パターンで、ePaper ディスプレイへの描画が正常に完了すること
- **SC-002**: 採用アルゴリズムにおける粒状感が現行と比較して目視で改善されていること（実機目視確認）
- **SC-003**: クリーンアップ後のコードに評価用の切り替えコードが残っていないこと
- **SC-004**: クリーンアップ後の表示品質がクリーンアップ前と同等であること（実機目視確認）

## Assumptions

- 評価は実機（ESP32S3 + ePaper ディスプレイ）での目視確認を基本とする
- アルゴリズムの切り替えはサーバー起動時の設定（環境変数またはコマンドライン引数）で行う
- 評価順序は B案 → A案 → A+B の順で実施する
- 最終採用アルゴリズムはユーザーが実機評価の結果を見て決定する
- 処理時間の大幅な増加（体感できるレベル）がない限り、画質を優先する

## Documentation Impact

- サーバーの画像変換処理に関するコメントを採用アルゴリズムに合わせて更新
