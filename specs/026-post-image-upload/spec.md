## Clarifications

- Q: 画像更新要求に認証を要求するか → A: 認証なしで受け付ける
- Q: 画像更新要求の受付 path はどれにするか → A: `/upload` に POST する

### User Story 1 - 新しい画像をアップロードしたい (Priority: P1)

運用者は外部クライアントから HTTP サーバへ画像を送信し、形式や解像度がそのままでは期待値と異なる場合でも、配信に使える `image.png` として更新したい。

**Why this priority**: 画像更新を手動ファイル配置に頼らず HTTP 経由で完了できることが、この機能の中核価値だから。

**Independent Test**: 既存画像がある状態で、新しい PNG 画像と PNG 以外の受理対象画像を raw body と multipart/form-data の両形式でそれぞれ POST し、保存結果が `image.png` に正規化され、480x800 への変換はアスペクト比維持の中央クロップで行われ、取得処理が更新後画像を入力として扱うことを確認する。

**Acceptance Scenarios**:

1. **Given** サーバが画像更新要求を受け付けられる状態, **When** 運用者が受理可能な画像を `/upload` へ raw body または multipart/form-data で POST する, **Then** サーバは必要に応じて PNG 形式へ正規化し、現在の `image.png` として保存して成功を返す。
2. **Given** 直前に画像更新が成功している状態, **When** 利用者が通常の画像取得を行う, **Then** サーバは更新後の `image.png` を入力として以後の配信結果を返す。

---

### User Story 2 - 不正なアップロードを判別したい (Priority: P2)

運用者は送信内容が壊れている、画像として扱えない、または要求形式が不正な場合に、更新されなかったことを判別したい。

**Why this priority**: 失敗時に既存の配信画像を壊さず、原因を切り分けられないと運用導線として危険だから。

**Independent Test**: 壊れたファイル、空の本文、画像として扱えない内容、または multipart/form-data 内に有効な画像ファイルが含まれない要求を POST し、既存の `image.png` が保持され、失敗理由が判別できることを確認する。

**Acceptance Scenarios**:

1. **Given** サーバが画像更新要求を受け付けている状態, **When** 運用者が画像として扱えない内容を POST する, **Then** サーバは保存を行わず、入力不正と判別できる失敗応答を返す。
2. **Given** 有効な `image.png` が既に存在する状態, **When** 無効なアップロード要求が送られる, **Then** サーバは既存の `image.png` を変更しない。

---

### User Story 3 - 更新結果をすぐ反映したい (Priority: P3)

運用者はサーバ再起動や手動同期をせずに、アップロード完了後すぐに、480x800 に整えられた更新済み画像を利用したい。

**Why this priority**: 更新のたびに追加作業が必要だと、HTTP 経由で更新できる利点が薄れるため。

**Independent Test**: サーバ起動中に解像度の異なる画像を含む複数回の画像を POST し、各回の直後に取得結果が最後に成功した 480x800 の保存済み画像へ切り替わることを確認する。

**Acceptance Scenarios**:

1. **Given** サーバが稼働中で画像更新が成功した状態, **When** 続けて画像取得が行われる, **Then** サーバは再起動なしで最新の保存済み `image.png` を使う。
2. **Given** 連続した更新要求が順に成功する状態, **When** 2 回目の画像更新が完了する, **Then** サーバは最後に成功した画像だけを現在の `image.png` として扱う。

---

### Edge Cases

- `image.png` がまだ存在しない初回状態でも、有効な画像更新要求 1 回で利用可能状態へ移行できること。
- アップロード途中で保存が完了しなかった場合、破損した `image.png` を配信対象として残さないこと。
- 本文が空、サイズ 0、画像として解釈できない、または multipart/form-data 内に保存対象画像が特定できない要求は成功扱いにしないこと。
- 受理可能な画像の解像度が 480x800 以外でも、保存時にはアスペクト比を維持した拡大縮小と中央クロップにより 480x800 の単一の現在画像へ正規化されること。
- 既存の画像取得経路は、画像更新機能追加後も従来どおり利用できること。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST 外部クライアントからの `POST /upload` により、raw PNG body と multipart/form-data の両形式で画像更新要求を受け付けられなければならない。
- **FR-002**: System MUST 有効な画像更新要求を受信したとき、その内容を PNG 形式の配信元入力画像 `image.png` として保存しなければならない。
- **FR-003**: Users MUST be able to 画像更新成功後、追加の手動ファイル操作なしで更新済み画像を配信対象として利用できなければならない。
- **FR-004**: System MUST 画像として扱えない内容、不完全な要求、multipart/form-data 内で保存対象画像を特定できない要求、または保存不能な要求に対して、保存を行わず失敗応答を返さなければならない。
- **FR-005**: System MUST 無効な更新要求が送られた場合でも、直前に利用可能だった `image.png` を破損または消失させてはならない。
- **FR-006**: System MUST 画像更新要求の成功時に、運用者が更新完了を判別できる応答を返さなければならない。
- **FR-007**: System MUST 画像更新機能の追加後も、既存の画像取得経路とその利用手順を維持しなければならない。
- **FR-008**: System MUST 連続した複数回の画像更新要求に対して、最後に成功した 1 件のみを現在の `image.png` として扱わなければならない。
- **FR-009**: System MUST 受理可能な画像形式で送られた更新要求に対して、保存前に PNG 形式へ正規化しなければならない。
- **FR-010**: System MUST 保存前の画像解像度が 480x800 でない場合、480x800 の現在画像になるようリサイズとクロップを行わなければならない。
- **FR-011**: System MUST 480x800 への正規化では、元画像のアスペクト比を維持したまま拡大縮小し、中央クロップを適用しなければならない。

### Key Entities *(include if feature involves data)*

- **画像更新要求**: 外部クライアントが送信する 1 回分の画像保存要求。raw PNG body または multipart/form-data のいずれかの形式、画像本文、要求の妥当性、受付結果を持つ。
- **現在画像**: サーバが配信や変換の入力として参照する単一の `image.png`。最後に成功した更新結果を表し、保存時点で PNG かつ 480x800 に正規化され、必要時はアスペクト比維持の中央クロップが適用されている。
- **更新結果応答**: `POST /upload` に対して返す成否通知。成功、入力不正、保存失敗などを判別するために用いる。

## Scope Boundaries *(mandatory)*

### Allowed Scope

- HTTP サーバへの画像更新用 POST 導線の追加
- `POST /upload` を用いた画像更新契約の追加
- 認証なしで受け付ける更新導線の明文化
- `image.png` の保存、形式正規化、480x800 へのアスペクト比維持リサイズと中央クロップ、置換、失敗時保護に関する挙動定義
- 既存の画像取得導線との整合確認
- この機能の利用手順と運用上の注意点の文書更新

### Forbidden Scope

- 480x800 への正規化を超える高度な自動編集、補正、加工仕様の追加
- 複数画像の管理、履歴保存、バージョン管理
- 認証認可基盤や外部ストレージ基盤の新規導入
- `firmware/` および `xiaozhi-esp32/` 側の変更

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 運用者は有効な画像更新要求 1 回で、現在画像を新しい内容へ置き換えられる。
- **SC-002**: 無効な画像更新要求に対して、運用者は 1 回の応答確認で「更新失敗」と判断でき、既存画像が維持されたことを確認できる。
- **SC-003**: 運用者は画像更新後にサーバ再起動や手動ファイル配置を行わず、次の画像取得から 480x800 に正規化され、中央クロップ規則が一貫した更新結果を利用できる。
- **SC-004**: 既存利用者は画像更新機能追加後も、従来の画像取得手順を変更せずに最新画像を取得できる。

## Assumptions

- 利用者はローカルネットワークまたは到達可能な外部クライアントから HTTP 経由で画像更新を行う。
- 画像更新要求は認証なしで受け付け、アクセス制御は今回の必須範囲に含めない。
- 更新対象は単一の現在画像のみであり、保存先ファイル名は引き続き `image.png` に固定する。
- 今回の入力画像は一般的な画像形式を受理対象とし、保存時には PNG 形式へ正規化する。
- 保存後の現在画像は 480x800 を基準サイズとし、元画像の解像度が異なる場合はアスペクト比を維持した拡大縮小と中央クロップで合わせる。
- 既存の取得経路は現在画像を入力として参照し続ける前提とする。

## Documentation Impact

- 画像更新用 POST 手順、成功時の確認方法、失敗時の切り分け方法を案内する文書更新が必要になる。
- 受理可能な入力形式、PNG への正規化、480x800 へのアスペクト比維持リサイズと中央クロップの挙動を利用手順へ追記する必要がある。
- 既存の `image.png` 手動配置手順がある場合、HTTP 経由更新を主導線として位置づけ直す必要がある。
- 既存の画像取得利用手順に、更新後の反映タイミングと運用上の注意点を追記する必要がある。
