# 機能仕様: HTTP アクセスログ追加

**Feature Branch**: `014-access-logs`  
**Created**: 2026-03-29  
**Status**: Draft  
**Input**: ユーザー記述: "httpサーバにアクセスログを追加"

**記述ルール**: この文書は日本語で記述する。固有名詞、コード識別子、ライブラリ名のみ原文維持可。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - リクエスト到達を確認したい (Priority: P1)

運用者は HTTP サーバへどのリクエストが到達したかをその場で確認し、サーバが実際にアクセスを受けているかを判断したい。

**Why this priority**: まず到達確認ができないと、配信失敗が入力画像やネットワークの問題なのか、そもそもリクエストが来ていないのか切り分けられないため。

**Independent Test**: サーバ起動中に `/` と `/image.bmp` へアクセスし、各リクエストごとにアクセスログが出力されることを確認する。

**Acceptance Scenarios**:

1. **Given** サーバが起動している状態, **When** 利用者が `/` にアクセスする, **Then** サーバは通常の応答に加えて 1 件のアクセスログを出力する。
2. **Given** サーバが起動している状態, **When** 利用者が `/image.bmp` にアクセスする, **Then** サーバは通常の応答に加えて 1 件のアクセスログを出力する。

---

### User Story 2 - 応答結果も把握したい (Priority: P2)

運用者はアクセスの有無だけでなく、そのリクエストが成功したのか失敗したのかもログで把握したい。

**Why this priority**: リクエストが来ていても、成功か失敗かが分からないと障害切り分けの価値が十分でないため。

**Independent Test**: 正常系と入力画像問題の失敗系をそれぞれ発生させ、アクセスログに応答結果が含まれることを確認する。

**Acceptance Scenarios**:

1. **Given** 正常に変換可能な入力画像がある状態, **When** 利用者が配信先へアクセスする, **Then** アクセスログには成功したことが分かる結果情報が含まれる。
2. **Given** 入力画像が未配置または変換不能な状態, **When** 利用者が配信先へアクセスする, **Then** アクセスログには失敗したことが分かる結果情報が含まれる。

---

### User Story 3 - ログで利用状況を追いたい (Priority: P3)

運用者は複数回のアクセスを見分けられるようにし、あとから簡単に利用状況を追いたい。

**Why this priority**: 単に同じ文言が並ぶだけでは、複数回のアクセスやクライアント差分を追いにくいため。

**Independent Test**: 異なるタイミングまたは異なるクライアント条件で複数回アクセスし、各アクセスが区別可能な形で記録されることを確認する。

**Acceptance Scenarios**:

1. **Given** サーバが起動している状態, **When** 利用者が複数回アクセスする, **Then** 各アクセスは個別のログ行として区別して確認できる。
2. **Given** クライアント情報が取得できる状態, **When** 利用者がアクセスする, **Then** ログからアクセス元や対象 path を読み取れる。

---

### Edge Cases

- 存在しない path へのアクセスでも、通常の失敗応答を壊さずにアクセスログが残ること。
- 入力画像未配置や変換不能などの失敗系でも、アクセスログ自体が欠落しないこと。
- 高頻度に連続アクセスがあっても、各リクエストが 1 件ずつ識別可能に記録されること。
- アクセスログ追加によって既存の BMP 配信内容や route 契約が変わらないこと。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST 各 HTTP リクエストに対してアクセスログを出力しなければならない。
- **FR-002**: System MUST アクセスログに、少なくともアクセス対象 path とリクエストごとの発生タイミングを含めなければならない。
- **FR-003**: System MUST アクセスログに、少なくとも成功か失敗かを判別できる応答結果情報を含めなければならない。
- **FR-004**: System MUST 取得可能な場合、アクセスログにアクセス元を判別できる情報を含めなければならない。
- **FR-005**: System MUST 正常系、入力画像未配置、変換不能、存在しない path へのアクセスのいずれでもアクセスログを残さなければならない。
- **FR-006**: System MUST アクセスログ追加によって既存の `/` と `/image.bmp` の配信契約を変更してはならない。
- **FR-007**: Users MUST be able to サーバ起動中の標準出力または既存の起動導線からアクセスログを確認できなければならない。

### Key Entities *(include if feature involves data)*

- **アクセスログ行**: 1 回の HTTP リクエストに対応する記録。発生タイミング、対象 path、応答結果、必要に応じてアクセス元情報を含む。
- **応答結果情報**: そのリクエストが成功したか失敗したかを示す記録要素。正常配信、入力画像未配置、変換不能、存在しない path などの判別に使う。

## Scope Boundaries *(mandatory)*

### Allowed Scope

- HTTP サーバのアクセスログ出力追加
- 正常系と失敗系の応答結果をログで区別できるようにする変更
- 利用手順や運用確認手順の更新

### Forbidden Scope

- 新しい HTTP route の追加
- 画像変換アルゴリズムや配信フォーマットそのものの変更
- 外部ログ基盤や永続ログ保管の導入
- `firmware/` や `xiaozhi-esp32/` 側の変更

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 運用者は 1 回のアクセスごとに 1 件のアクセスログを確認できる。
- **SC-002**: 運用者はログを見るだけで、対象 path と成功/失敗を 1 行以内で判別できる。
- **SC-003**: 運用者は正常系と入力画像問題の失敗系を、追加のデバッグ操作なしでログから切り分けられる。
- **SC-004**: アクセスログ追加後も、既存クライアントは同じ取得先で従来どおり BMP 応答または既存の失敗応答を受け取れる。

## Assumptions

- アクセスログはまずローカル開発時や手元運用時に確認できる簡潔なサーバログとして出力されればよい。
- 既存サーバは標準出力で起動案内を出しており、アクセスログも同じ確認導線で扱うのが自然である。
- 今回はログ保存先の追加やローテーションまでは要求しない。

## Documentation Impact

- サーバ起動後にアクセスログをどう確認するかの手順更新が必要になる。
- 正常系と失敗系でどのようなログが出るかを運用文書へ反映する必要がある。
