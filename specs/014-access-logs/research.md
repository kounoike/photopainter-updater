# Research: HTTP アクセスログ追加

## Decision 1: アクセスログは標準出力へ 1 リクエスト 1 行で出す

- Decision: アクセスログは既存の起動案内と同じ標準出力へ出し、各 HTTP リクエストごとに 1 行で完結させる。
- Rationale: 今回の要件はまず手元運用で到達確認と切り分けができることにある。外部ログ基盤やファイル保存を増やさず、既存導線のまま確認できる構成が最小である。
- Alternatives considered:
  - ファイルへ永続保存する: スコープ外であり運用複雑化が大きいため不採用。
  - 外部ログ基盤へ送る: ローカル優先方針に反するため不採用。

## Decision 2: ログ項目は時刻・アクセス元・method・path・status を最小セットとする

- Decision: 各アクセスログ行には少なくとも時刻、アクセス元、HTTP method、対象 path、応答ステータスを含める。
- Rationale: spec の FR-002、FR-003、FR-004 を満たしつつ、1 行で読める最低限の切り分け情報になる。method を含めると 404 や将来の拡張時にも文脈が崩れにくい。
- Alternatives considered:
  - path と status だけにする: 発生タイミングやアクセス元が分からず運用価値が不足するため不採用。
  - user-agent などを常に出す: 今回は要求されておらず冗長になりやすいため不採用。

## Decision 3: 正常系だけでなく失敗系と未定義 path も同じ導線で記録する

- Decision: `/` と `/image.bmp` の成功応答だけでなく、入力画像未配置、変換不能、存在しない path へのアクセスもアクセスログに残す。
- Rationale: 切り分け価値は失敗系で高い。特に「リクエストが来ていない」のか「来て失敗した」のかを見分けるため、失敗系も同じ仕組みで記録する必要がある。
- Alternatives considered:
  - 成功系だけログを出す: 障害時に最も必要な情報が抜けるため不採用。
  - 失敗系は別形式の詳細ログに分ける: 1 行で読める一貫性が崩れるため不採用。

## Decision 4: 実装は既存レスポンス構築の直前または直後で共通出力する

- Decision: 既存のレスポンス生成処理に近い場所で、request 情報と response 情報が揃った時点で共通ログ出力を行う。
- Rationale: middleware を増やさなくても、現在の単純な route 構成なら十分対応できる。小さい変更で route 契約を維持しやすい。
- Alternatives considered:
  - 専用 middleware を導入する: 将来的には有効だが、今回の scope に対しては実装コストが上がるため不採用。
