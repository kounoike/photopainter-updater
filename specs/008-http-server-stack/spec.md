# 機能仕様: HTTPサーバ技術選定調査

**Feature Branch**: `008-http-server-stack`  
**Created**: 2026-03-29  
**Status**: Draft  
**Input**: ユーザー記述: "HTTPサーバ機能の基本的な枠組みを作る前に技術検討を行う。Rust+axumとPython+FastAPIのどっちが良いか。他の案はあるか？"

**記述ルール**: この文書は日本語で記述する。固有名詞、コード識別子、ライブラリ名のみ原文維持可。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 実装前に有力候補を比較したい (Priority: P1)

開発者は HTTP サーバ機能の実装に着手する前に、有力な技術候補を比較し、どの候補がこのプロジェクトに最も適するかを判断したい。

**Why this priority**: サーバ基盤の選択は後続実装の構成、保守性、運用の単純さに大きく影響するため、最初に判断軸を固定する価値が高い。

**Independent Test**: 少なくとも Rust+axum、Python+FastAPI、その他候補 1 つ以上について、比較観点と採用判断が文書化され、読み手が採用理由を説明できることを確認する。

**Acceptance Scenarios**:

1. **Given** HTTP サーバの基本機能をまだ実装していない状態, **When** 技術調査を行う, **Then** 開発者は複数候補を同じ観点で比較した記録を得られる。
2. **Given** 比較結果がまとまった状態, **When** 後続 feature の planning に進む, **Then** 開発者は採用候補または第一候補を根拠付きで参照できる。

---

### User Story 2 - ローカル運用に合う選択をしたい (Priority: P2)

開発者は PhotoPainter Updater の LAN 内運用やシンプルな保守に合う構成を優先し、過剰な複雑さを避けたい。

**Why this priority**: このプロジェクトはローカル優先・運用単純性が重要であり、一般論だけで選ぶと実情に合わない可能性があるため。

**Independent Test**: 比較結果に、ローカル運用、開発体験、保守性、依存の重さ、将来の画像前処理拡張への適合性が含まれていることを確認する。

**Acceptance Scenarios**:

1. **Given** 技術候補を比較する状態, **When** 評価観点を定義する, **Then** 調査結果にはこのプロジェクト固有の運用前提が反映される。

---

### User Story 3 - 画像処理と telemetry まで含めて比較したい (Priority: P3)

開発者は HTTP サーバが将来的に担う画像前処理とデバイス情報収集まで見据えて、それぞれの技術候補で実現可能かを判断したい。

**Why this priority**: サーバの本命責務が画像前処理や監視連携に広がるなら、単純な静的配信だけで技術選定すると判断を誤るため。

**Independent Test**: 各候補について、フルカラー画像入力、回転・スケーリング・ディザリング・6 色インデックス化、デバイス側 POST 受信、監視連携の観点が評価結果に含まれていることを確認する。

**Acceptance Scenarios**:

1. **Given** 画像前処理と監視連携を想定する状態, **When** 技術調査を行う, **Then** 開発者は各候補でその機能群がどの程度実現しやすいかを比較できる。

---

### User Story 4 - 見送る候補の理由も残したい (Priority: P4)

開発者は採用しなかった候補についても簡潔な理由を残し、同じ比較を将来やり直さずに済むようにしたい。

**Why this priority**: 調査だけで終わる場合でも、却下理由が残っていないと次の feature で判断がぶれやすくなるため。

**Independent Test**: 調査結果に、採用候補だけでなく見送り候補とその理由が残り、次の feature が再比較なしで前進できることを確認する。

**Acceptance Scenarios**:

1. **Given** 複数候補を比較した状態, **When** 技術選定結果をまとめる, **Then** 採用候補と見送り候補の理由が明確に記録される。

### Edge Cases

- Rust+axum と Python+FastAPI の差が決定的でない場合でも、次の実装 feature に進める暫定第一候補または追加調査条件を残すこと。
- 他候補が最有力になる場合でも、比較結果が Rust+axum / Python+FastAPI 中心の依頼意図を無視しないこと。
- 調査の結果、現時点では実装着手より先にサーバ要件の分解が必要と分かった場合、その理由を記録すること。
- 画像入力は BMP 前提ではなく、元画像として PNG などのフルカラー画像を受ける想定で比較すること。
- 画像処理では `ref/convert.py` が示すディザリング、回転、スケーリング、6 色インデックス化の流れを参考要件として扱うこと。
- デバイス情報 POST 受信は、将来の Grafana 監視や低バッテリー通知へつながる前提で比較すること。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST Rust+axum と Python+FastAPI を比較対象として含む技術調査結果を残さなければならない。
- **FR-002**: System MUST Rust+axum と Python+FastAPI 以外に、少なくとも 1 つの代替候補または「代替候補なし」の妥当な理由を記録しなければならない。
- **FR-003**: System MUST 比較結果に、ローカル運用適合性、開発体験、保守性、依存の重さ、将来拡張性の評価観点を含めなければならない。
- **FR-004**: System MUST 後続の HTTP サーバ feature が参照できる採用候補または第一候補を明示しなければならない。
- **FR-005**: System MUST 見送った候補についても、簡潔で再利用可能な理由を記録しなければならない。
- **FR-006**: System MUST 利用者向け文書ではなく、開発文書として技術選定の判断根拠を残さなければならない。
- **FR-007**: System MUST 各候補について、PNG などのフルカラー画像入力を前提にした回転、スケーリング、ディザリング、6 色インデックス化の実現性を評価しなければならない。
- **FR-008**: System MUST `ref/convert.py` を参考実装として扱い、その処理内容を各候補でどの程度再現しやすいかを比較しなければならない。
- **FR-009**: System MUST 各候補について、デバイスからの POST による telemetry 受信、時系列蓄積、Grafana 監視、閾値通知の実現しやすさを評価しなければならない。

### Key Entities *(include if feature involves data)*

- **技術候補**: HTTP サーバの基本構成候補。比較対象、長所、短所、採否を持つ。
- **評価観点**: 候補を比較するための判断軸。ローカル運用、保守性、拡張性などを含む。
- **技術選定結果**: 第一候補、代替候補、見送り理由をまとめた記録。
- **画像前処理要件**: フルカラー入力を回転、スケーリング、ディザリング、6 色インデックス化してデバイス向け出力に変換する要件。
- **デバイス telemetry 要件**: バッテリー残量などのデバイス情報を POST 受信し、監視や通知へ接続する要件。

## Scope Boundaries *(mandatory)*

### Allowed Scope

- HTTP サーバ技術候補の比較調査
- 比較結果の設計文書・調査文書への反映
- 後続 feature が参照できる判断材料の明文化
- 画像前処理と telemetry 収集の観点を含めた比較

### Forbidden Scope

- `server/` や新規サーバ実装コードの追加
- `firmware/` の HTTP クライアント実装変更
- 実際の API 設計やエンドポイント実装
- `xiaozhi-esp32/` 配下の直接変更

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Rust+axum、Python+FastAPI、その他候補 1 つ以上について、同一文書内で比較と採否が確認できる。
- **SC-002**: 後続 feature の planning 時に、追加の技術比較を行わなくても第一候補を参照できる。
- **SC-003**: 見送り候補の理由が文書化され、読み手が再比較の必要性を即判断できる。
- **SC-004**: 調査結果がローカル運用・最小構成の前提と矛盾していない。
- **SC-005**: 各候補について、画像前処理要件と telemetry 要件の実現性が同一比較枠で確認できる。

## Assumptions

- 今回の目的は実装ではなく技術選定調査である。
- HTTP サーバは将来的に画像配信や前処理の足場になる可能性がある。
- 現時点では LAN 内運用と保守の単純さを優先する。
- 元画像はフルカラー画像（PNG など）を想定する。
- telemetry の初期想定はバッテリー残量送信であり、監視系への接続は後続 feature で具体化する。
