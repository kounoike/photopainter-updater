# リサーチ: ディザリング向け画像改善アイデア整理

**Branch**: `019-dither-image-ideas` | **Date**: 2026-03-30

## 1. 改善候補の実装単位

**Decision:** 改善候補は前処理とディザリング設定を束ねた named profile として表現する。

**Rationale:** 既存実装は `DitherOptions` に複数フラグを持っており、個別フラグをさらに増やすと比較条件の把握が難しくなる。profile 化すれば、実験実行時に「どの改善案を試したか」を一意に扱え、結果記録と比較表を同期しやすい。

**Alternatives considered:**
- 個別の環境変数を都度増やす: 組み合わせ爆発が起きやすく、比較結果の再現性が落ちる。
- ハードコードで案を切り替える: 実験の切り戻しと比較がしにくい。

---

## 2. 比較実験の入力画像戦略

**Decision:** 固定画像セットを比較の基準にし、必要時のみ追加画像を補足として使う。

**Rationale:** 実機 ePaper 表示を主判定にする以上、同じ画像群で繰り返し比較できることが重要になる。固定セットを基準にすると、profile 間の差分を一定条件で見られ、追加画像は特定の懸念点を掘る補助用途に限定できる。

**Alternatives considered:**
- 毎回自由な画像を使う: 実験条件が揺れて比較が成立しにくい。
- 固定画像だけに限定する: 実運用で気になる画像パターンへの追試がしにくい。

---

## 3. 実験の操作インターフェース

**Decision:** 既存の環境変数読込と `server/run.sh` を拡張し、HTTP ルートは変更しない。

**Rationale:** 現行サーバーは `ServerConfig::from_env()` と `run.sh` を入口にしており、ここを拡張するのが最小変更で済む。firmware は既存の `image.bmp` / `image.bin` を取得できればよく、配信経路を増やす必要がない。

**Alternatives considered:**
- 新しい HTTP パラメータやルートを追加する: 配信面の回帰確認が増え、firmware 境界にも影響しやすい。
- 設定ファイルを新設する: 実験のたびに編集対象が増え、現行運用より複雑になる。

---

## 4. 実験結果の記録方法

**Decision:** 比較結果は `specs/019-dither-image-ideas/` 配下の文書成果物として残し、主判定は実機 ePaper 表示、補助情報として `cargo test` と生成画像確認を使う。

**Rationale:** この feature の価値は「改善案を実際に試し、比較し、次に進める候補を絞ること」にある。実装コードだけでは判断根拠が残らないため、profile、画像セット、観点、判定、次アクションを文書として残す必要がある。

**Alternatives considered:**
- 実験結果をコミットメッセージや口頭メモに任せる: 再利用しにくく、tasks や review に繋がらない。
- PC 上の変換結果だけで判定する: ePaper 特性を主目的にしている今回の feature とずれる。

---

## 5. 実装スコープの境界

**Decision:** 実装変更は server 側に限定し、firmware、HTTP エンドポイント、転送フォーマットは現状維持とする。

**Rationale:** 現状でも server が画像変換の責務を持っており、firmware は受信・表示に専念している。今回の比較対象は画像加工品質なので、server-only に閉じるのが責務分離と最小変更の両方に合う。

**Alternatives considered:**
- firmware に比較補助や表示切り替え機能を足す: scope が広がり、feature 本体の評価軸から外れる。
- 転送フォーマットを変えて評価情報を埋め込む: 禁止範囲に抵触し、比較実験には不要。
