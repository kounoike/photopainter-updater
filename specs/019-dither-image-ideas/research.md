# リサーチ: ディザリング向け画像改善アイデア整理

**Branch**: `019-dither-image-ideas` | **Date**: 2026-03-30

## 1. 改善候補の実装単位

**Decision:** 改善候補は前処理とディザリング設定を束ねた named profile として表現する。

**Rationale:** 既存実装は `DitherOptions` に複数フラグを持っており、個別フラグをさらに増やすと比較条件の把握が難しくなる。profile 化すれば、実験実行時に「どの改善案を試したか」を一意に扱え、結果記録と比較表を同期しやすい。

**Alternatives considered:**
- 個別の環境変数を都度増やす: 組み合わせ爆発が起きやすく、比較結果の再現性が落ちる。
- ハードコードで案を切り替える: 実験の切り戻しと比較がしにくい。

---

## 2. 比較実験の入力画像戦略

**Decision:** 同じ入力画像を手動で差し替えながら比較し、必要時のみ追加画像を補足として使う。

**Rationale:** 実機 ePaper 表示を主判定にする以上、同じ入力画像に対する差分を直接見比べられることが重要になる。単一画像の手動差し替えなら現行運用に近く、profile 間の差分を一定条件で見られ、別画像は特定の懸念点を掘る補助用途に限定できる。

**Alternatives considered:**
- 毎回自由な画像を使う: 実験条件が揺れて比較が成立しにくい。
- 複数画像セットを自動巡回する: 今回の scope に対して実装が重い。

---

## 3. 実験の操作インターフェース

**Decision:** 既存の環境変数読込と `server/run.sh` を拡張し、HTTP ルートは変更しない。

**Rationale:** 現行サーバーは `ServerConfig::from_env()` と `run.sh` を入口にしており、ここを拡張するのが最小変更で済む。firmware は既存の `image.bmp` / `image.bin` を取得できればよく、配信経路を増やす必要がない。

**Alternatives considered:**
- 新しい HTTP パラメータやルートを追加する: 配信面の回帰確認が増え、firmware 境界にも影響しやすい。
- 設定ファイルを新設する: 実験のたびに編集対象が増え、現行運用より複雑になる。

---

## 4. 実験結果の記録方法

**Decision:** 比較結果は `specs/019-dither-image-ideas/` 配下の文書成果物として残し、主判定は実機 ePaper 表示、補助情報として `cargo test` と生成画像確認を使う。

**Rationale:** この feature の価値は「改善案を実際に試し、比較し、次に進める候補を絞ること」にある。実装コードだけでは判断根拠が残らないため、profile、入力画像、比較モード、観点、判定、次アクションを文書として残す必要がある。

**Alternatives considered:**
- 実験結果をコミットメッセージや口頭メモに任せる: 再利用しにくく、tasks や review に繋がらない。
- PC 上の変換結果だけで判定する: ePaper 特性を主目的にしている今回の feature とずれる。

---

## 5. 実装スコープの境界

**Decision:** 実装変更は server 側に限定し、firmware、HTTP エンドポイント、転送フォーマットは現状維持とする。

**Rationale:** 現状でも server が画像変換の責務を持っており、firmware は受信・表示に専念している。今回の比較対象は画像加工品質なので、server-only に閉じるのが責務分離と最小変更の両方に合う。

**Alternatives considered:**
- firmware に比較補助や表示切り替え機能を足す: scope が広がり、feature 本体の評価軸から外れる。
- 転送フォーマットを変えて評価情報を埋め込む: 禁止範囲に抵触し、比較実験には不要。

---

## 6. 改善仮説メモ

以下は現時点の改善アイデアの整理であり、最終決定ではない。`tasks` と `implement` では、ここから比較対象 profile を絞って実験する。

### 6-1. 問題の切り分け

**Observation:** 色味が弱く見える問題には、デバイス起因の限界と変換処理起因の悪化が混在している。

- ePaper は非発光デバイスであり、有色表示そのものの発色上限が低い
- したがって「鮮やかな発色そのもの」を作ることには限界がある
- 一方で、量子化やディザリングの選択次第で、不要に白黒へ吸われたり、不自然な差し色が混ざったりして、見え方をさらに悪化させる可能性がある

**Implication:** 今回の改善目標は「発色限界の克服」ではなく、「変換処理による不要な色消失と不自然な色置換の抑制」に置く。

### 6-2. 彩度ブーストの再評価

**Hypothesis:** 既存の彩度ブーストは ePaper の弱い発色を前処理で無理に補う対症療法であり、本質的な解決ではない可能性が高い。

- 彩度を持ち上げると、一部の画像では見え方が改善する可能性はある
- ただし、副作用として不自然な発色、色相の崩れ、局所的な嘘っぽさを招きやすい
- そのため、改善の主戦場は前処理よりも「候補色選択」と「誤差拡散の使い方」に寄せるべきという仮説を置く

### 6-3. 白黒と有色の採用バランス

**Hypothesis:** 色味の弱さの一因は、最近傍色選択で白・黒が選ばれやすすぎることにあるかもしれない。

- 白黒 2 色と有色パレットで採用頻度に差をつけることで、有色が不必要に埋もれるのを防げる可能性がある
- ただし、有色を優先しすぎると画面全体が濁ったり、偽色が増えたりする
- そのため、単純な「有色優先」ではなく、白黒/有色の採用コストを条件付きで変える方向が有望

**Candidate direction:** パレット候補選択時に、白・黒・有色それぞれへ重み付けまたはペナルティを導入する。

### 6-4. 色相ガード

**Hypothesis:** 大きな誤差を許しても色相的に不自然な色が入らないよう、色相差に基づく制約が必要。

- 例として、ピンク系の領域に青が入り込むような置換は避けたい
- 単純な RGB 距離や Lab 距離だけでは、明度や総距離の都合で色相の飛びが起こりうる
- そのため、候補色選択時に色相差ペナルティを追加し、ありえない飛び色を抑える方針を検討する

**Candidate direction:** 候補色スコアを `色距離 + 色相差ペナルティ + パレット重み補正` のような合成評価にする。

### 6-5. 白の使い方

**Hypothesis:** 白を増やせば色が強くなるわけではないが、明部や抜けを作る用途では有効かもしれない。

- 白の混在によって、有色の面が相対的に見やすくなる場面はある
- ただし、白が多すぎると有色面積が減り、かえって色が弱く見える
- そのため、白は「色を強くする色」ではなく、「見え方のコントラストを整える補助色」として扱う方がよい

### 6-6. 黒の使い方

**Hypothesis:** 黒を的確に入れることで輪郭や面の差分が際立ち、有色が相対的に見えやすくなる可能性がある。

- 黒は輪郭を締め、隣接する有色面を認識しやすくする
- 一方で、黒を入れすぎると画面が汚れ、中間調や柔らかい面が潰れる
- そのため、黒は全面的に増やすのではなく、「境界」や「暗部」で効かせる候補として扱う

**Candidate direction:** 有色を守る戦略と、輪郭・暗部で黒を効かせる戦略を分けて比較する。

### 6-7. 輪郭情報の補助利用

**Hypothesis:** Canny のような輪郭抽出を直接出力に焼き込むのではなく、黒の使いどころを決める補助情報として使える可能性がある。

- 輪郭が強い場所では黒を使う価値が高い
- 輪郭が弱い場所では黒を増やしすぎない方がよい
- そのため、輪郭マップをディザリング時の重み付けに使う設計は有望

**Notes:**
- 手法は Canny 固定にせず、Sobel や局所コントラスト量も比較候補に含める
- 輪郭抽出そのものが目的ではなく、「輪郭情報をどう候補色選択に反映するか」を評価対象にする

### 6-8. 現時点の比較候補

当面の profile 候補は次のように整理できる。

1. 彩度ブーストを弱める、または撤廃する
2. 白黒と有色の採用重みを変える
3. 色相ガードを導入する
4. 輪郭情報を使って黒の優先度を調整する
5. 2 と 3 の組み合わせ
6. 2 と 4 の組み合わせ

実装初期に試す profile は上記 catalog から 2-4 件へ絞るが、catalog 自体は 5 件以上を維持して比較記録の母集団として残す。

### 6-9. 比較観点の補強

既存の比較観点に加え、以下を意識して記録する。

- 有色が不必要に白黒へ吸われていないか
- 色相的に不自然な飛び色が減っているか
- 白が抜けとして効いているか、それとも色を痩せさせているか
- 黒が輪郭補助として効いているか、それとも画面を汚しているか
- 写真、肌色、空、植物、線画で挙動が変わらないか

### 6-10. 比較表示の進め方

**Hypothesis:** 全画面を別々に比較するより、1 枚の画像の中で左右または上下に処理違いを並べた方が、実機での差分を一度に評価しやすい可能性がある。

- 全画面 A と全画面 B を別々に見ると、表示タイミングや観察時の印象差が入りやすい
- 1 枚の中で A/B を分割すれば、同じ画像内容、同じ表示条件のまま差分を比較できる
- 特に、色味の残り方、輪郭の締まり、飛び色の有無は split view の方が気づきやすい可能性がある

**Candidate direction:**
- 左右 2 分割で baseline と比較対象 profile を同時表示する
- 必要に応じて上下分割も試し、画像内容によって見やすい方を選ぶ
- split view 自体が境界アーティファクトを作る可能性があるため、最終判断前に全画面表示でも再確認する
