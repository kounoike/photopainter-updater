# 機能仕様: デフォルト画像フォーマットを .bin に変更

**Feature Branch**: `016-default-bin-format`
**Created**: 2026-03-30
**Status**: Draft
**Input**: ユーザー記述: "今、/のルートはbmpを返しているが、これをimage.binと同じルートになるよう切り替えてデフォルトを.binにしたい。ファーム側も.bmpが無い限り.binとして扱うようにする"

**記述ルール**: この文書は日本語で記述する。固有名詞、コード識別子、ライブラリ名のみ原文維持可。

## Clarifications

### Session 2026-03-30

- Q: サーバーの `/` ルートにおける BMP フォールバックの意図（FR-003）は何か？ → A: サーバーはソース画像（PNG/JPEG）をオンザフライで変換するため「`.bmp` ファイルのみ存在する」シナリオは発生しない。FR-003 はスコープ外として削除する。`/` は常にバイナリを返し、BMP が必要なクライアントは `/image.bmp` を使用する。
- Q: SC-002「表示成功率」の意味と検証方法は何か？ → A: 「成功率」は不適切。実質パス/フェイルの確認であり、「ePaper ディスプレイへの描画が正常に完了すること（実機目視確認）」に修正する。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - ルートエンドポイントが .bin を返す (Priority: P1)

サーバーの `/` ルートにアクセスしたとき、現在は BMP 形式で画像が返却されているが、変更後は `image.bin` ルートと同等の `.bin` 形式で返却されるようになる。

**Why this priority**: これがこの機能の中核変更であり、他の変更の前提となるため最優先。

**Independent Test**: サーバーを起動し、`/` ルートにリクエストを送信して、レスポンスが `.bin` 形式であることを確認する。

**Acceptance Scenarios**:

1. **Given** サーバーが起動中で、ストレージに画像ファイルが存在する, **When** クライアントが `/` エンドポイントにリクエストを送信する, **Then** サーバーは `.bin` 形式で画像データを返す

---

### User Story 2 - ファームウェアがデフォルトで .bin を扱う (Priority: P1)

ファームウェアが画像データを受信・処理する際、リクエスト URL が明示的に `.bmp` で終わらない限り、データを `.bin` 形式として扱うように動作を変更する。

**Why this priority**: サーバー側の変更と整合性を取るために同時に対応が必要。

**Independent Test**: ファームウェアに `.bin` 形式の画像データを送信し、正しく表示・処理されることを確認する。URL が `.bmp` で終わる場合は従来通り BMP として処理されることも確認する。

**Acceptance Scenarios**:

1. **Given** ファームウェアが起動中で、`image_url` が `.bmp` で終わらない, **When** サーバーから画像データを取得する, **Then** ファームウェアはデータを `.bin` 形式として正しく処理・表示する
2. **Given** ファームウェアが起動中で、`image_url` が `.bmp` で終わる, **When** サーバーから画像データを取得する, **Then** ファームウェアはデータを BMP 形式として処理する
3. **Given** ファームウェアが起動中で、`image_url` が `.bmp` で終わらない, **When** サーバーから画像データを取得する, **Then** ファームウェアはデータを `.bin` 形式として処理する

---

### User Story 3 - 既存の image.bin ルートとの整合性 (Priority: P2)

`/` ルートと `/image.bin` ルートが同一の動作をするようになり、クライアントはどちらのエンドポイントを使っても同じ結果を得られる。

**Why this priority**: 既存クライアントとの後方互換性を維持しつつ、ルーティングを統一する。

**Independent Test**: `/` と `/image.bin` の両方にリクエストを送り、同じ形式・同じデータが返ることを確認する。

**Acceptance Scenarios**:

1. **Given** サーバーが起動中である, **When** クライアントが `/` と `/image.bin` それぞれにリクエストを送信する, **Then** 両方のエンドポイントが同一形式・同一内容のレスポンスを返す

---

### Edge Cases

- ストレージに画像ファイルが存在しない場合、適切なエラーレスポンスを返す
- ファームウェアが旧バージョンのサーバー（BMP を返す）に接続した場合の互換性は本機能のスコープ外とする

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST `/` ルートへのリクエストに対して `.bin` 形式で画像データを返す（現在の BMP 返却から変更）
- **FR-002**: System MUST `/` ルートの動作を `image.bin` ルートと同等にする
- **FR-004**: System MUST ファームウェアが URL サフィックスに基づいてフォーマットを判定できるようにする（`.bmp` で終わる場合のみ BMP 扱い、それ以外はバイナリ）
- **FR-005**: System MUST `image_url` が `.bmp` で終わらない場合、ファームウェアはデータを `.bin` 形式として処理する
- **FR-006**: System MUST `image_url` が `.bmp` で終わる場合、ファームウェアは引き続き BMP 形式として処理する

## Scope Boundaries *(mandatory)*

### Allowed Scope

- サーバー側の `/` ルートの返却フォーマット変更（BMP → `.bin`）
- ファームウェア側の画像フォーマット判定ロジック変更（デフォルトを `.bin` に）
- 既存の `image.bin` ルートとの動作統合

### Forbidden Scope

- `image.bin` ルート自体の挙動変更
- BMP 以外の画像フォーマット（PNG, JPEG 等）への対応
- 画像変換処理の追加
- 認証・認可の変更
- クライアント側 UI の変更

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: `/` ルートへのリクエストが `.bin` 形式のデータを返す割合が 100%
- **SC-002**: ファームウェアが `.bin` 形式の画像データを受信した際、ePaper ディスプレイへの描画が正常に完了すること（実機目視確認）
- **SC-003**: `/` ルートと `/image.bin` ルートのレスポンス形式が 100% 一致する
- **SC-004**: `image_url` が `.bmp` で終わる環境でのファームウェア動作が変更前と同等であること（後方互換性 100% 維持）

## Assumptions

- 現在の `image.bin` ルートは正常に動作しており、その実装が `/` ルートの参照実装となる
- サーバーはソース画像（PNG/JPEG）をオンザフライで変換するため、`.bmp` ファイルのみ存在するシナリオは発生しない
- 既存の BMP を使用しているクライアントは存在しないか、または `.bin` 形式への移行が許容される
- ターゲットデバイスは ESP32S3 を前提とする

## Documentation Impact

- サーバー API ドキュメントの `/` ルート説明を更新（BMP → `.bin`）
- ファームウェアのフォーマット判定ロジックに関するコメントを更新
