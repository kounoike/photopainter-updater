# 機能仕様: run.sh 配信設定改善

**Feature Branch**: `012-fix-run-access-path`  
**Created**: 2026-03-29  
**Status**: Draft  
**Input**: ユーザー記述: "run.shで起動したとき、2つの問題点がある。127.0.0.1:8000にbindしているので、0.0.0.0:8000にしたい。server/contentsディレクトリを可変にして、どこでrun.shを実行しても良いようにしたい。この2つを対応する"

**記述ルール**: この文書は日本語で記述する。固有名詞、コード識別子、ライブラリ名のみ原文維持可。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - LAN 端末から配信へアクセスしたい (Priority: P1)

利用者は `run.sh` で起動した配信先に、同一ネットワーク上の別端末から到達したい。

**Why this priority**: 画像取得側の機器が別端末である前提では、ローカルホスト限定の待受だと機能価値を満たせないため。

**Independent Test**: サーバ起動後、同一ネットワーク上の別端末から配信 URL へアクセスし、期待するファイルを取得できることを確認する。

**Acceptance Scenarios**:

1. **Given** 利用者が `run.sh` で配信を起動できる状態, **When** サーバを起動する, **Then** 配信はローカルホスト限定ではなくネットワーク外部から接続可能な待受状態になる。
2. **Given** サーバが起動している, **When** 同一ネットワーク上の別端末から配信 URL にアクセスする, **Then** サーバは要求されたファイルを返す。

---

### User Story 2 - 配信元ディレクトリを切り替えたい (Priority: P2)

利用者は用途に応じて配信元ディレクトリを指定し、既定の配置に依存せずにファイルを配信したい。

**Why this priority**: 配信内容の差し替えや一時ディレクトリの利用を簡単にしないと、`server/contents/` 固定運用がボトルネックになるため。

**Independent Test**: 既定以外のディレクトリを指定して起動し、そのディレクトリの内容が配信されることを確認する。

**Acceptance Scenarios**:

1. **Given** 利用者が配信したいファイル群を任意のディレクトリに用意している, **When** そのディレクトリを指定して `run.sh` を起動する, **Then** サーバは指定ディレクトリの内容を配信する。

---

### User Story 3 - 実行場所に依存せず起動したい (Priority: P3)

利用者はリポジトリ直下以外の任意のカレントディレクトリからでも、同じ方法で `run.sh` を起動したい。

**Why this priority**: 実行場所に制約があると、スクリプト利用手順が不安定になり自動化や運用手順へ組み込みにくいため。

**Independent Test**: リポジトリ外または別ディレクトリをカレントディレクトリにした状態で `run.sh` を起動し、期待どおり配信されることを確認する。

**Acceptance Scenarios**:

1. **Given** 利用者のカレントディレクトリがリポジトリ直下ではない, **When** `run.sh` を起動する, **Then** サーバは起動元ディレクトリに影響されず必要な配信設定を解決して起動する。

---

### Edge Cases

- 指定された配信元ディレクトリが存在しない場合、サーバは起動せず、利用者が誤りを判別できる案内を返すこと。
- 指定された配信元ディレクトリが空でも、サーバは異常終了せず、空ディレクトリとして扱うか内容不足として判別可能な応答を返すこと。
- 既定の配信元ディレクトリを使う場合でも、起動方法は追加設定なしで維持されること。
- 起動時の案内は、ローカル確認用の URL だけでなく別端末からの利用を誤解させない内容であること。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST `run.sh` 起動時の待受先をローカルホスト限定にせず、同一ネットワーク上の外部端末から接続可能な状態で配信しなければならない。
- **FR-002**: System MUST 起動時の案内に、外部端末から利用する際に必要な接続先を利用者が判別できる情報を含めなければならない。
- **FR-003**: Users MUST be able to 起動時に配信元ディレクトリを明示的に指定できなければならない。
- **FR-004**: System MUST 配信元ディレクトリが明示的に指定されない場合、既存の標準ディレクトリを既定値として利用できなければならない。
- **FR-005**: System MUST 配信元ディレクトリの解決をカレントディレクトリに依存させず、どの場所から起動しても同じ結果にならなければならない。
- **FR-006**: System MUST 指定された配信元ディレクトリが利用できない場合、起動を中止し、原因を利用者が判断できる失敗理由を表示しなければならない。
- **FR-007**: System MUST 既存の基本起動手順を維持し、標準ディレクトリを使う利用者に追加の必須設定を要求してはならない。

### Key Entities *(include if feature involves data)*

- **配信元ディレクトリ指定**: 起動時に解決される配信対象ディレクトリ。既定値または利用者指定値を取りうる。
- **公開アクセス先**: サーバ起動後に利用者や別端末が参照する接続先情報。待受状態と案内表示の整合対象となる。

## Scope Boundaries *(mandatory)*

### Allowed Scope

- `run.sh` の起動条件、既定値、利用者向け案内の見直し
- 配信元ディレクトリ指定の導線追加
- この変更に伴う利用手順や関連文書の更新

### Forbidden Scope

- サーバの配信対象ファイル形式や endpoint の追加
- 画像変換や生成機能の追加
- `firmware/` や `xiaozhi-esp32/` 側の仕様変更

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 利用者は `run.sh` を起動した後、同一ネットワーク上の別端末から 1 回のアクセスで配信内容を取得できる。
- **SC-002**: 利用者は標準ディレクトリを使う場合、従来どおり追加設定なしで配信を開始できる。
- **SC-003**: 利用者は任意の配信元ディレクトリを指定して起動し、その内容が反映された配信結果を確認できる。
- **SC-004**: 利用者はリポジトリ外や別作業ディレクトリから起動しても、手順変更なしで同じ配信結果を再現できる。

## Assumptions

- 利用者はローカルネットワーク内で配信確認を行う。
- 現在の標準配信元は `server/contents/` であり、互換性維持のため既定値として残す。
- 配信元ディレクトリの可変化は、起動時指定で満たせればよく、永続設定管理までは今回の必須範囲に含めない。

## Documentation Impact

- `run.sh` の起動方法と配信元ディレクトリ指定方法を案内する文書更新が必要になる。
- 外部端末からの接続確認方法が分かる利用手順の追記が必要になる。
