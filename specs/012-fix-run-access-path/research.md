# Research: run.sh 配信設定改善

## Decision 1: 待受先の改善は `run.sh` の起動案内と環境受け渡しで吸収する

- Decision: サーバの待受先は既存どおり外部到達可能な構成を維持し、`run.sh` とサーバ起動時の案内表示を「localhost 限定」に見えない内容へ統一する。
- Rationale: 現行の `server/src/main.rs` はすでに外部到達可能な待受を行っている一方、`run.sh` と起動ログが `127.0.0.1` 前提の説明を出している。要求を満たすために必要なのは bind 実装の全面変更ではなく、利用者が正しい接続先を認識できる起動導線の整備である。
- Alternatives considered:
  - サーバ本体だけを変更する: 実際の誤解ポイントである `run.sh` の案内が残り、利用者価値を満たしにくいため不採用。
  - 新しい設定ファイルで待受先を管理する: 今回のスコープでは複雑化が大きく、既定利用者の手順も増えるため不採用。

## Decision 2: 配信元ディレクトリは起動時指定を優先し、既定値は `server/contents/` を維持する

- Decision: 配信元ディレクトリは起動時に明示指定できるようにし、未指定時のみ `server/contents/` を既定値として使う。
- Rationale: spec の FR-003、FR-004、FR-007 を最小構成で同時に満たせる。既定利用者は今までどおり起動でき、必要なときだけ任意ディレクトリへ切り替えられる。
- Alternatives considered:
  - 常にカレントディレクトリを配信元にする: 既存互換性が壊れ、誤起動のリスクも高いため不採用。
  - 永続設定ファイルを追加する: 今回の変更規模に対して大きすぎるため不採用。

## Decision 3: パス解決はスクリプト基準で絶対パス化し、起動元カレントディレクトリの影響を排除する

- Decision: `run.sh` は自身の配置場所を基準に既定値を求め、利用者指定の配信元も起動前に絶対パスとして検証してからサーバへ渡す。
- Rationale: どこから `run.sh` を実行しても同じ結果になるという FR-005 を直接満たせる。Bash で完結でき、Rust 側へ不要な複雑さを持ち込まない。
- Alternatives considered:
  - Rust サーバ側でのみ相対パスを解決する: 起動前エラーの分かりやすさや `run.sh` の責務整理が弱くなるため不採用。
  - 利用者に絶対パスのみを要求する: 手順が不便になり、既定利用者との整合も悪いため不採用。

## Decision 4: 外部インターフェース契約は HTTP route ではなく `run.sh` の起動インターフェースとして定義する

- Decision: 今回の contract はサーバ API ではなく、`run.sh` の起動方法、既定値、失敗条件、利用者向け出力を対象にする。
- Rationale: feature が変更する対象は HTTP route ではなく起動インターフェースであるため、契約はその境界に合わせる方が tasks と検証へ直結する。
- Alternatives considered:
  - 既存の HTTP contract を再定義する: 変更対象とずれ、重複が増えるため不採用。
