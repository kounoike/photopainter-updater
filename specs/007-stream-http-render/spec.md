# 機能仕様: HTTP画像の直接表示検討

**Feature Branch**: `007-stream-http-render`  
**Created**: 2026-03-29  
**Status**: Draft  
**Input**: ユーザー記述: "HTTPダウンロードした画像をSDカードに保存せずに直接書き換えられないか検討し、可能そうなら実装する"

**記述ルール**: この文書は日本語で記述する。固有名詞、コード識別子、ライブラリ名のみ原文維持可。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - SDカードへの一時保存なしで更新したい (Priority: P1)

利用者は起動時または BOOT ボタン更新時に、取得画像を SD カードへ一時保存しなくても e-paper の表示更新が完了し、従来と同じように最新画像を見られることを期待する。

**Why this priority**: 画像更新のたびに SD カードへ一時書き込みを行わない構成にできれば、書き込み負荷や一時ファイル依存を減らせる可能性があるため。

**Independent Test**: 正常な設定と取得可能な画像を用意し、更新後に e-paper 表示が更新されること、かつ SD カード上に更新用一時画像ファイルが生成・更新されないことを確認する。

**Acceptance Scenarios**:

1. **Given** 正常な設定と取得可能な画像がある状態, **When** 起動時更新または BOOT ボタン更新が実行される, **Then** デバイスは SD カードへの一時画像保存を必須とせずに e-paper 表示を更新できる。
2. **Given** 直接表示方式が採用された状態, **When** 更新が完了する, **Then** 利用者は従来と同等の表示結果を確認できる。

---

### User Story 2 - 既存の失敗判定を壊したくない (Priority: P2)

利用者または開発者は、保存方式を変えても設定不備、通信失敗、画像不正の判定や終了挙動が従来どおり維持されることを期待する。

**Why this priority**: 一時保存をなくしても、既存の運用上重要な失敗切り分けや終了動作が壊れると価値よりリスクが大きくなるため。

**Independent Test**: 設定不備、HTTP 失敗、不正画像を個別に発生させ、直接表示方式でも従来どおり失敗分類と終了挙動が維持されることを確認する。

**Acceptance Scenarios**:

1. **Given** 更新開始後に画像取得失敗または画像不正が発生する状態, **When** デバイスが更新処理を進める, **Then** デバイスは成功扱いにせず、既存と同じ粒度で失敗終了する。
2. **Given** 直接表示方式を採用した状態, **When** 更新が失敗する, **Then** 利用者または開発者は従来どおり失敗理由を判断できる。

---

### User Story 3 - 不可能なら理由を明確にしたい (Priority: P3)

開発者は、直接表示方式が安全に成立しない場合でも、なぜ採用できないかを明確に残し、既存方式を維持した上で判断できるようにしたい。

**Why this priority**: この変更は実現可能性調査を含むため、無理な場合でも理由が曖昧だと将来同じ調査を繰り返すことになるため。

**Independent Test**: 検討結果として、直接表示方式を採用するか、既存方式を維持するかのどちらかが文書と手動確認結果で説明できることを確認する。

**Acceptance Scenarios**:

1. **Given** 直接表示方式が安全条件を満たせない状態, **When** 実装判断を行う, **Then** 既存の SD カード一時保存方式を維持しつつ、採用しなかった理由が文書に残る。
2. **Given** 直接表示方式が採用可能な状態, **When** 実装が完了する, **Then** 既存方式との差分と運用上の変化が利用者向け文書に反映される。

### Edge Cases

- HTTP 応答が途中で途切れた場合でも、部分受信データで表示を更新しないこと。
- 画像ヘッダ検証に必要な情報が揃う前に不正が判明した場合、既存と同じく画像不正として終了すること。
- 表示更新中に BOOT ボタンが重複押下されても、更新ジョブが多重化しないこと。
- 直接表示方式がメモリ制約で成立しない場合、既存方式へ安全に戻せること。
- 既存の `download.bmp` を利用した診断や運用がある場合、その変更有無が文書で明確に説明されること。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST HTTP 取得画像を SD カードへ一時保存せずに表示更新できるかを検討し、採用可否を明確に判断できる状態を残さなければならない。
- **FR-002**: System MUST 直接表示方式が安全に成立する場合、起動時更新と BOOT ボタン更新の両方で、一時画像ファイルへの依存をなくした更新方式を提供しなければならない。
- **FR-003**: System MUST 直接表示方式を採用しても、利用者が従来と同等の表示結果を確認できるようにしなければならない。
- **FR-004**: System MUST 保存方式の変更後も、設定不備、通信失敗、画像不正の失敗分類と終了挙動を維持しなければならない。
- **FR-005**: System MUST 直接表示方式が安全条件を満たせない場合、既存の SD カード一時保存方式を維持し、その理由を開発文書に残さなければならない。
- **FR-006**: System MUST 利用者向け文書で、更新時に一時画像ファイルが使われるかどうかを説明しなければならない。

### Key Entities *(include if feature involves data)*

- **更新画像受信データ**: HTTP 応答から順次受け取る画像データ。表示更新に使えるかどうかの判断対象になる。
- **更新方式**: 一時保存方式または直接表示方式のどちらを採用するかという実装上の選択結果。
- **表示更新結果**: 利用者が e-paper 上で確認する最終表示状態。従来と同等であることが求められる。
- **検討結果記録**: 直接表示方式を採用した、または採用しなかった理由を残す記録。

## Scope Boundaries *(mandatory)*

### Allowed Scope

- `firmware/` 配下での HTTP 取得後の画像処理経路の見直し
- 直接表示方式の可否判断に必要な検証と実装
- 既存の失敗分類、終了動作、更新トリガーの維持確認
- 利用者向け文書および開発文書への判断結果の反映

### Forbidden Scope

- 画像取得元や設定ファイル形式の変更
- `xiaozhi-esp32/` 配下の直接改変
- e-paper の表示仕様や画像フォーマット要件そのものの拡張
- 直接表示検討と無関係な WiFi、電源、ボタン仕様の変更

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 正常な更新手動確認で、採用された更新方式が起動時更新と BOOT ボタン更新の両方で 60 秒以内に従来同等の表示結果を出せる。
- **SC-002**: 直接表示方式を採用した場合、更新後に SD カード上の一時画像ファイル更新が発生しないことを確認できる。
- **SC-003**: 設定不備、HTTP 失敗、画像不正の各確認で、保存方式変更後も従来どおり失敗分類と終了挙動を確認できる。
- **SC-004**: 直接表示方式を採用しない場合でも、採用見送り理由と現行方式維持の判断が文書で説明されている。

## Assumptions

- 対象は既存の `firmware/` 実装であり、起動時更新と BOOT ボタン更新の基本仕様は維持する前提とする。
- 画像は引き続き 24-bit BMP を前提とし、表示要件は変更しない。
- 調査の結果、直接表示方式が安全でない場合は現行方式維持を許容する。
- 利用者にとって重要なのは表示結果と運用手順であり、内部保存方式の変更はその価値を補助するものとみなす。
