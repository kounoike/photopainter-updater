# 機能仕様: 写真調ディザリングの追加改善

**Feature Branch**: `020-adaptive-diffusion-tuning`  
**Created**: 2026-03-30  
**Status**: Draft

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 次の改善案を選べる (Priority: P1)

画像変換品質を改善したい担当者として、`019` の比較結果で残った写真調画像の弱点に対して、どの追加改善案を試すべきかを整理された形で確認したい。これにより、場当たり的な調整ではなく、青保持と淡色面保持に効く具体策へ絞って進められる。

**Why this priority**: 次に試す改善案が曖昧なまま実装を始めると、比較の軸がぶれて `019` の結果を活かせないため。

**Independent Test**: 成果物を読み、追加改善案の狙い、比較対象、優先順位、見送る案が区別できることを確認する。

**Acceptance Scenarios**:

1. **Given** `019` の結論を引き継ぐ feature 成果物を確認している, **When** 改善案一覧を読む, **Then** 青保持、明るい低彩度面、肌の中間調保持に対する追加改善案の狙いと優先度が整理されている
2. **Given** 複数の追加改善案が存在する, **When** 範囲を確認する, **Then** 今回実装する案と、比較対象または後回しにする案が区別できる

---

### User Story 2 - 追加改善を試せる (Priority: P2)

評価担当者として、写真調画像で問題になっていた青空、淡色面、肌の中間調に対して、追加改善をサーバー側で実際に試し、既存上位候補と比較したい。これにより、改善案が本当に前進かどうかを同じ運用導線で判断できる。

**Why this priority**: 改善案の整理だけでは価値が足りず、実際に画像変換へ反映して比較できなければ採否判断に使えないため。

**Independent Test**: 代表的な写真調画像を入力として、既存上位候補と追加改善案の出力を比較し、ローカル検証と手動評価の両方で差分を説明できることを確認する。

**Acceptance Scenarios**:

1. **Given** サーバーが起動できる状態である, **When** 追加改善案を選んで画像変換を実行する, **Then** 既存の運用方法を大きく変えずに改善後の出力を取得できる
2. **Given** 青空や明るい低彩度面を含む写真調画像がある, **When** 既存上位候補と追加改善案を比較する, **Then** 差分を確認できる比較結果が残る

---

### User Story 3 - 採用判断へ進める (Priority: P3)

実装判断を行う担当者として、追加改善の結果から、既存上位候補を置き換えるか、補強するか、保留するかを判断したい。これにより、次の採用判断や既定値見直しへ進める。

**Why this priority**: 今回の feature は試作で終わらせず、`019` の暫定上位候補を次の段階へ進める判断材料を作ることが目的だから。

**Independent Test**: 成果物を読み、今回の追加改善が `advance` / `hold` / `reject` のどれに当たるかと、その理由を説明できることを確認する。

**Acceptance Scenarios**:

1. **Given** 比較結果が揃っている, **When** 判定結果を確認する, **Then** 既存上位候補との関係と次アクションが明記されている
2. **Given** 改善量が限定的または条件付きの案がある, **When** 結論を確認する, **Then** 保留または後続課題として扱う理由が残っている

### Edge Cases

- 青空を含む画像では改善しても、肌や低彩度背景を悪化させる案は採用候補として扱わないこと
- 低彩度面の保持を優先した結果、黒粒や偽色が増える場合は副作用として明示すること
- 追加 fixture が写真カテゴリに偏りすぎる場合でも、少なくとも既存のイラスト調評価を壊さない前提で比較すること
- 既存の `color-priority + DITHER_DIFFUSION_RATE=0.8` を下回る案は、試せても既定採用候補へ昇格させないこと

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST `019` の比較結果を引き継ぎ、今回追加で改善したい視覚課題を青保持、明るい低彩度面、肌の中間調保持の観点で明文化する
- **FR-002**: System MUST 今回試す追加改善案を少なくとも 1 件定義し、既存上位候補との差分と狙いを説明できるようにする
- **FR-003**: Users MUST 既存の server 側起動導線から追加改善案を選択または比較できる
- **FR-004**: System MUST 青空または青系の広い面を含む写真調画像を比較入力に加え、青保持の確認結果を残せるようにする
- **FR-005**: System MUST 明るい低彩度面と肌の中間調に関する評価観点を比較結果へ含める
- **FR-006**: System MUST 追加改善案のローカル回帰確認手段を用意し、既存の配信経路、転送フォーマット、firmware を変更せずに検証できるようにする
- **FR-007**: System MUST 比較結果に基づき、追加改善案を `advance` / `hold` / `reject` のいずれかに分類し、次アクションを記録する

### Key Entities *(include if feature involves data)*

- **追加改善案**: `019` の上位候補を補強するために新たに試す画像変換方針。狙う課題、比較対象、副作用、判定結果を持つ。
- **評価画像セット**: 青空、明るい低彩度面、肌の中間調など、今回の弱点を確認するために使う代表入力画像群。用途と観察したい観点を持つ。
- **比較結果**: 既存上位候補と追加改善案を同条件で見比べた記録。観察点、良かった点、悪化した点、判定、次アクションを持つ。

## Scope Boundaries *(mandatory)*

### Allowed Scope

- `server/` 配下の画像変換ロジック改善
- 追加改善案の設定、比較方法、評価画像、手順文書の更新
- `specs/020-adaptive-diffusion-tuning/` 配下の成果物作成
- `019` の結論を踏まえた比較結果と採否判断の追記

### Forbidden Scope

- `firmware/` 配下の実装変更
- `xiaozhi-esp32/` 配下の参照コード変更
- HTTP エンドポイントや転送フォーマットの変更
- ePaper 実機以外の新しい評価基盤や外部サービスの導入

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 関係者が成果物を読み、今回の追加改善案と既存上位候補の差分、狙い、採用しない案を 5 分以内に説明できる
- **SC-002**: 青空または青系の広い面を含む写真調画像を含め、少なくとも 2 種類の代表画像で比較結果が残っている
- **SC-003**: 追加改善案について、青保持、明るい低彩度面、肌の中間調のうち少なくとも 1 つで前進かどうかを判断できる
- **SC-004**: ローカル回帰確認を通したうえで、既存の配信経路や firmware を変更せずに改善案を試せる

## Assumptions

- `019` で暫定上位候補となった `color-priority + DITHER_DIFFUSION_RATE=0.8` は今回の比較基準として維持する
- 改善の主戦場は `server/` 側の画像変換であり、配信経路や表示側の責務分離は変えない
- 青保持の評価には、既存の `image7` とは別に青系の広い面を含む写真調画像が必要になる
- 実機主判定は引き続き有効だが、今回の実装ではローカルで再現できる比較と回帰確認も必須とする
