## User Scenarios & Testing *(mandatory)*

### User Story 1 - 少数画像で LoRA 試作を実行する (Priority: P1)

開発者として、少数の参照画像だけを用意して、ローカル環境でキャラクター LoRA の試作学習を実行したい。これにより、クラウド前提の大規模学習に進む前に、手元環境でキャラクター再現の成立性を確認できる。

**Why this priority**: 少数画像と手元 GPU で学習を開始できなければ、この feature 全体の検証が始められないため。

**Independent Test**: 必要最小限の参照画像セットと定義済みの手順を使って試作学習を開始し、学習成果物が出力されれば完了。

**Acceptance Scenarios**:

1. **Given** 少数の参照画像が用意されている, **When** 試作学習の手順を実行する, **Then** 学習ジョブが途中で前提不足にならず開始される
2. **Given** 試作学習が完了した, **When** 出力先を確認する, **Then** 後続の生成で利用できる学習成果物が確認できる

---

### User Story 2 - 少数画像向けのデータ準備方針を決める (Priority: P2)

開発者として、どのような参照画像を何枚程度そろえれば試作に進めるかを把握したい。これにより、画像収集コストを抑えつつ、キャラクターの恒常特徴を学習しやすいデータセットを用意できる。

**Why this priority**: 学習以前にデータ準備が曖昧だと、試作が成功しても再現性がなく、次のキャラクターへ横展開しにくいため。

**Independent Test**: データ準備ルールに従って最小構成の画像セットを作り、試作学習に投入できれば完了。

**Acceptance Scenarios**:

1. **Given** 新しいキャラクターで試作したい, **When** データ準備方針を参照する, **Then** 必要な画像の種類、避けるべき画像、軽い前処理の範囲が判断できる
2. **Given** 背景やポーズの影響を抑えたい, **When** データ準備方針を参照する, **Then** キャラクター本体の恒常特徴を優先するための基準が分かる

---

### User Story 3 - 将来の自動生成フローへ接続可能にする (Priority: P3)

開発者として、試作したキャラクター LoRA を将来的に ComfyUI の全自動画像生成フローへ組み込みたい。これにより、毎回参照画像を解釈してキャラクター差し替えするのではなく、学習済みキャラクター表現を前提に季節・イベント・天気に応じた画像生成へ発展できる。既存の参照画像ベース差し替え方式では、ポーズ保持、構図保持、キャラクター恒常特徴保持を同時に安定させることが難しかったため、本 feature ではキャラクター恒常特徴を事前学習済み表現として扱う方向へ寄せる。なお今回のスコープは、将来統合の前提整理と学習成果物の最小限の再利用確認までに留める。

**Why this priority**: 学習試作だけ成功しても、将来の生成フローへ接続できない構成では運用価値が限定されるため。

**Independent Test**: 学習成果物の利用前提、入力条件、既存生成フローとの接続点が成果物に整理されていれば完了。

**Acceptance Scenarios**:

1. **Given** 試作学習で得たキャラクター LoRA がある, **When** 運用想定を確認する, **Then** 既存の自然言語ベース生成フローへどのように組み込むか理解できる
2. **Given** 将来イベント衣装分岐を導入したい, **When** 設計方針を確認する, **Then** 通常衣装維持とイベント衣装切替の発展余地が残っている

### Edge Cases

- 参照画像が少なすぎる、または顔・髪・衣装が十分に見えない場合でも、試作不成立の原因が切り分けられること
- ローカル環境の GPU メモリ不足で学習条件を満たせない場合でも、縮退した試作条件や制約が把握できること
- 参照画像の背景やポーズの癖が強い場合でも、キャラクター本体の恒常特徴を優先するための判断基準があること
- 試作 LoRA の品質が本番運用には不十分でも、成立性確認として継続可否を判断できること

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST 少数の参照画像からキャラクター LoRA 試作学習を開始できる手順と実行条件を定義する
- **FR-002**: System MUST VRAM 12GB 程度のローカル環境を前提に、量子化や offload を活用した現実的な試作条件を提示する
- **FR-003**: Users MUST be able to 参照画像の枚数、画像内容、許容する軽い前処理の範囲を基準に、試作用データセットを準備できる
- **FR-004**: System MUST キャラクターの恒常特徴を優先して学習させるための caption 方針、データ準備方針、必要な補助情報の扱いを定義する
- **FR-005**: System MUST 試作学習の成果物、保存先、再利用前提を整理し、後続の生成フローで利用可能な単位として扱えるようにする
- **FR-006**: System MUST 試作学習の成立可否を判断するための最小評価観点を定義し、少数画像での再現性確認方法を示す
- **FR-007**: System MUST 既存の自然言語ベース生成フローへ将来統合するために、学習成果物と生成時入力の責務分界を整理する
- **FR-008**: System MUST 通常衣装維持とイベント衣装切替の両方へ将来拡張できるよう、キャラクター恒常特徴と衣装可変要素を分けて扱う方針を定義する

### Key Entities *(include if feature involves data)*

- **参照画像セット**: キャラクター LoRA の試作学習に使う少数画像群。顔、髪、衣装、付属特徴などの恒常要素が読み取れる必要がある。
- **キャラクター LoRA 試作成果物**: 少数画像から学習して得られる再利用可能なキャラクター表現。後続の自動生成フローで参照される。
- **学習構成**: ローカル GPU 制約下で試作を成立させるための入力条件、実行条件、評価観点をまとめた運用単位。
- **生成統合前提**: 学習済みキャラクター表現を既存の自然言語ベース生成フローへ接続する際の責務分界と運用ルール。

## Scope Boundaries *(mandatory)*

### Allowed Scope

- `specs/024-zimage-character-lora/` 配下の成果物作成
- 少数画像向けキャラクター LoRA 試作学習の方針整理
- ローカル GPU 制約下での試作条件、データ準備条件、評価条件の整理
- 学習成果物の最小限の再利用確認
- 将来の ComfyUI 自動生成フローへの統合前提の整理

### Forbidden Scope

- firmware や `xiaozhi-esp32/` 配下の変更
- 本 feature での完成度の高い本番学習やクラウド学習基盤の構築
- 個別キャラクター向けの手動レタッチや都度 inpaint を前提にした運用
- 既存の HTTP サーバや ComfyUI ワークフローの全面的な実装置換
- ComfyUI の全自動生成フローへの本格統合実装

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 開発者が 1 キャラクター分の少数参照画像セットを準備し、半日以内に試作学習を開始できる
- **SC-002**: 試作学習の結果として、同一キャラクターとして識別できる画像を 1 つ以上再生成できるかを判断できる
- **SC-003**: 成果物を読めば、少数画像試作に必要な画像条件、学習前提、評価観点を誤解なく説明できる
- **SC-004**: 後続の生成フロー設計者が、学習済みキャラクター表現を自然言語ベース生成へ統合する際の責務分界を説明できる

## Assumptions

- ローカルの主対象環境は VRAM 12GB 程度の単一 GPU を想定する
- 今回は本番品質の最適化ではなく、少数画像での成立性確認を主目的とする
- 今回は学習試作基盤と学習成果物の最小限の再利用確認までを対象とし、ComfyUI への本格統合は後続 feature で扱う
- キャラクターの恒常特徴は、毎回の参照画像解釈より学習済み表現で扱う方針とする
- 参照画像ベースの差し替え方式は、ポーズ保持とキャラクター保持の綱引きになりやすく、全自動運用の前提としては不安定だったとみなす
- 個別の崩れ箇所に対する手動 inpaint や個別プロンプト調整を前提にしない
- 季節、イベント、天気などのシーン可変要素は、学習済みキャラクター表現とは別に後続フローで扱う
