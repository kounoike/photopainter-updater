# Research: HTTPサーバ構成整理

## Decision 1: 環境変数の読込は `envconfig` に集約する

- Decision: `PORT`、`CONTENT_DIR`、ディザ設定など現在 `main.rs` で個別に読んでいる起動設定は `envconfig` の 1 つの設定構造体へ集約し、既定値、型変換、入力検証の責務を起動コードから分離する。
- Rationale: 既存コードでは環境変数の取得方式が値ごとに分散しており、未指定時の既定値適用と不正値時の案内規則が追いにくい。設定を 1 箇所へ集めると、運用者向け案内と単体テストの観点を同じモデルで扱える。
- Alternatives considered:
  - `std::env` のまま helper 関数だけ増やす: 分散は多少減るが、設定定義の一覧性と入力規則の一元化が弱い。
  - CLI 引数パーサへ全面移行する: 今回の起動導線は `server/run.sh` と環境変数中心であり、ユーザー価値に対して変更量が大きい。
  - 手書きの設定パーサを新設する: 依存は減るが、既定値・型変換・テストの定型実装を自前で持つことになる。

## Decision 2: 運用ログは `tracing` 系で統一する

- Decision: 起動ログ、設定要約、アクセスログ、失敗時の案内は `println!` ベースの出力から `tracing` と `tracing-subscriber` に寄せ、同じ初期化経路で整形・レベル管理できるようにする。
- Rationale: 現在は起動メッセージとアクセスログがそれぞれ独自の出力経路を持ち、将来のログ追加時に表記揺れや初期化重複が起きやすい。`tracing` で統一すれば、起動時とリクエスト時の可観測性を揃えつつ、テストではイベント生成箇所を明確に保てる。
- Alternatives considered:
  - `println!` を維持して文言だけ揃える: 短期的には最小変更だが、ログ責務の整理という目的に対して伸びしろが小さい。
  - `log` クレートと別 backend を使う: 実現可能だが、構造化フィールドや今後の拡張性を考えると `tracing` の方が整合しやすい。
  - 外部ログ基盤を導入する: Forbidden Scope に抵触し、今回のローカル優先方針に合わない。

## Decision 3: モジュール境界は入出力責務で切る

- Decision: ファイル分割は「設定」「起動配線」「route/handler」「ログ」「応答生成」「画像パイプライン」の責務で切り、`main.rs` はサーバ起動の配線だけを持つ構成とする。
- Rationale: 既存 `main.rs` は 1500 行超で、同じファイル内に純粋関数、HTTP handler、起動処理、テスト helper が混在している。入出力責務ごとの分割にすると、変更時に追う依存方向が明確になり、P1 の保守性要求と P2/P3 の運用要求を同時に満たしやすい。
- Alternatives considered:
  - 単純に前半/後半へ分割する: 行数は減るが責務分離にならず、将来の変更点特定が難しい。
  - 画像処理だけ外部 crate 化する: 再利用性は上がるが、今回の scope に対して過剰で依存境界も増える。
  - router と handler のみ分ける: HTTP 層だけは整理できるが、設定とログの肥大化が残る。

## Decision 4: 外部契約は既存 route と起動導線を固定する

- Decision: 新しい route や起動方法は追加せず、`/`、`/image.bmp`、`/image.bin`、`server/run.sh`、既存の環境変数を維持したまま内部整理を行う。
- Rationale: 今回の feature は内部保守性の改善であり、利用者や firmware から見える契約を変えると回帰の切り分けが難しくなる。外部契約を固定し、contract と quickstart へ明文化することで、リファクタリングの安全性を高められる。
- Alternatives considered:
  - route 名や設定名を整理のついでに変更する: 設計の見た目は整うが、互換維持要求に反する。
  - `run.sh` を廃止して cargo 直起動へ統一する: シンプルではあるが、既存の運用導線を壊す。
  - Binary/BMP 経路の表現を変更する: 今回の外部価値ではなく、別 feature として扱うべき。

## Decision 5: テストは責務ごとの回帰に再編する

- Decision: 既存の `main.rs` 内テストは、設定、HTTP route、画像応答、ログ出力の責務単位で配置し直し、少なくとも `cargo test` で既定値起動、設定エラー、主要 route 応答が継続確認できる構成を採る。
- Rationale: リファクタリングでは外部挙動を保ちながら内部構造だけが変わるため、責務単位のテストがないと差分の安全性を説明しづらい。既存テスト資産を活かしつつ再配置すれば、SC-004 の「独立した受け入れ確認」に直結する。
- Alternatives considered:
  - テストはそのまま `main.rs` に残す: 実装分割後の責務境界と一致せず、保守性向上の効果が薄い。
  - 手動確認だけに寄せる: 変更頻度が上がる構成整理には回帰検知が足りない。
  - スナップショット中心に置き換える: ログや文面の変更には有効だが、今回の主要価値は route と設定の挙動維持であり過不足がある。
