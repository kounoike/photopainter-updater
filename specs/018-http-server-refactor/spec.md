## User Scenarios & Testing *(mandatory)*

### User Story 1 - 変更しやすい構成で保守したい (Priority: P1)

開発者は HTTP サーバの責務が整理された構成で作業し、画像変換、配信、設定読込、ログ出力などの変更を安全に行いたい。

**Why this priority**: 単一ファイルへの責務集中は変更時の影響範囲が読みにくく、今後の機能追加や不具合修正の速度と安全性を下げるため。

**Independent Test**: サーバコードの主要責務が分離された状態で、画像配信と既存の変換処理が従来どおり動作することを確認する。

**Acceptance Scenarios**:

1. **Given** 既存の HTTP サーバ機能が存在する状態, **When** 開発者が配信処理を確認する, **Then** 配信処理の責務が他の起動処理や設定読込と分離され、変更対象を局所化して追跡できる。
2. **Given** 既存の HTTP サーバ機能が存在する状態, **When** 開発者が画像変換処理を確認する, **Then** 画像変換に関する責務が他の運用処理と分離され、挙動を独立して理解できる。

---

### User Story 2 - 起動設定を一貫して扱いたい (Priority: P2)

運用者はサーバ起動時の設定値を一貫した方法で扱い、設定不足や不正値がある場合も原因をすぐ把握したい。

**Why this priority**: 設定取得や既定値適用の規則が散らばると、起動失敗時の判断が難しくなり、運用ミスが起きやすくなるため。

**Independent Test**: 既定値起動、不正な設定値、明示的な設定指定の各条件でサーバを起動し、挙動と案内が一貫していることを確認する。

**Acceptance Scenarios**:

1. **Given** 必須でない設定が未指定の状態, **When** 運用者がサーバを起動する, **Then** サーバは既定値を用いて一貫した方法で起動する。
2. **Given** 不正な設定値が渡された状態, **When** 運用者がサーバを起動する, **Then** サーバは不正箇所を判別できる案内を返し、原因を特定できる。

---

### User Story 3 - 実行状況を読みやすく確認したい (Priority: P3)

運用者はサーバ起動時とリクエスト処理時の情報を読みやすく確認し、通常運用時の状態把握と障害切り分けをしやすくしたい。

**Why this priority**: 構成整理だけでは運用価値が不足し、起動や配信時の状況を一貫して追えないと保守性改善の効果が限定されるため。

**Independent Test**: サーバ起動とリクエスト処理を実行し、主要な実行状況が一貫した形式で確認できることを確かめる。

**Acceptance Scenarios**:

1. **Given** サーバが起動する状態, **When** 運用者が起動ログを確認する, **Then** 起動に必要な設定内容と待受状態を読みやすく把握できる。
2. **Given** サーバがリクエストを処理する状態, **When** 運用者が実行ログを確認する, **Then** 正常系と失敗系の違いを同じ確認導線で把握できる。

### Edge Cases

- 既定値で起動できる環境と明示設定が必要な環境のどちらでも、設定読込の規則が変わらず理解できること。
- 不正な設定値が 1 つだけ存在する場合でも、他の正常な設定と混同せずに原因を特定できること。
- 構成整理後も既存の配信先、応答形式、画像変換結果の利用者視点の契約を壊さないこと。
- 失敗系のログ追加や設定整理によって、正常系の配信確認手順が複雑化しないこと。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST HTTP サーバの主要責務を、少なくとも起動設定、HTTP 経路定義、画像生成または変換、運用ログ出力の単位で分離し、単一ファイルへ過度に集中させない。
- **FR-002**: System MUST 既存の HTTP 配信先と応答契約を維持したまま、構成整理後も既存利用者が同じ手順で画像取得できるようにしなければならない。
- **FR-003**: System MUST 起動設定を一貫した規則で読み取り、未指定時は既定値を適用し、不正値時は原因が判別できる案内を返さなければならない。
- **FR-004**: System MUST 起動時に、運用者が待受状態と主要設定を把握できる実行情報を出力しなければならない。
- **FR-005**: System MUST リクエスト処理時に、運用者が正常系と失敗系を区別できる実行情報を一貫した形式で確認できるようにしなければならない。
- **FR-006**: System MUST 構成整理後も、既存の画像変換および配信の受け入れ確認を責務単位で検証できる状態を提供しなければならない。
- **FR-007**: Users MUST be able to 配信処理、設定読込、ログ出力のいずれかを変更する際、無関係な責務を追わずに対象箇所を特定できなければならない。

### Key Entities *(include if feature involves data)*

- **起動設定**: サーバ起動時に読み取る入力値の集合。待受先、コンテンツ参照先、変換動作に関わる既定値と検証結果を含む。
- **サーバ責務単位**: 配信、変換、設定、ログ出力など、変更対象を分離して扱うための論理的なまとまり。
- **実行ログ**: 起動時およびリクエスト処理時に出力される運用向け情報。待受状態、設定要約、処理結果、失敗原因の識別に用いる。

## Scope Boundaries *(mandatory)*

### Allowed Scope

- HTTP サーバ内部構成の整理とファイル分割
- 起動設定の読込規則と既定値適用の整理
- 起動時および処理時の運用向け出力改善
- 既存機能を維持するための受け入れ確認および関連文書更新

### Forbidden Scope

- 新しい HTTP API や配信フォーマットの追加
- 画像変換アルゴリズム自体の仕様変更
- `firmware/` および `xiaozhi-esp32/` 側の変更
- 外部監視基盤、永続ログ基盤、設定管理サービスの新規導入

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 開発者はサーバの主要責務を 1 つの巨大ファイルではなく責務単位で追跡でき、対象責務の特定に 5 分以上の横断調査を要しない。
- **SC-002**: 運用者は既定値起動、不正設定、正常リクエスト、失敗リクエストの各ケースで、原因または状態を 1 つの確認導線で判断できる。
- **SC-003**: 既存クライアントは構成整理後も同じ配信先から従来どおり画像を取得でき、利用手順の追加変更を必要としない。
- **SC-004**: 保守担当者は設定読込、ログ出力、画像配信の各責務について、独立した受け入れ確認を実施して回帰有無を判断できる。

## Assumptions

- 対象は既存の HTTP サーバ実装であり、既存の配信挙動は維持する前提とする。
- 今回の主目的は保守性と運用性の改善であり、外部仕様の拡張は求められていない。
- 起動設定は環境ごとの差異を吸収できるよう、既定値と入力検証を備えた一貫した扱いが必要である。

## Documentation Impact

- サーバの起動方法と設定項目の説明を、整理後の構成に合わせて更新する必要がある。
- 運用時に確認すべき起動情報と実行ログの見方を文書化する必要がある。
- 保守担当者向けに、責務分割後の変更対象の探し方や確認観点を反映する必要がある。
