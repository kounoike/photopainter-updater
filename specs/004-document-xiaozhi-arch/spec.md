## User Scenarios & Testing *(mandatory)*

### User Story 1 - 全体構造を短時間で把握できる (Priority: P1)

開発者は `xiaozhi-esp32` 配下を一通り読まなくても、主要ディレクトリ、中心となるコンポーネント、起動から動作開始までの責務分担に加え、代表的な主要フローを技術文書から把握できる。

**Why this priority**: 構造の全体像が見えなければ、既存コードの改修判断や後続の設計作業が毎回ゼロからの読解になり、開発効率が大きく低下するため。

**Independent Test**: 初見の開発者が技術文書だけを読んで、主要ディレクトリの役割と実行経路の概要を説明できることを確認する。

**Acceptance Scenarios**:

1. **Given** 開発者が `xiaozhi-esp32` の実装を未読の状態で、 **When** 作成された技術文書を読む, **Then** 主要ディレクトリと中核モジュールの責務を把握できる。
2. **Given** 開発者が起動処理の入口を知りたい状態で, **When** 技術文書の構成説明を参照する, **Then** エントリポイントから起動・通信・音声を含む代表的な主要フローまで追跡できる。

---

### User Story 2 - 関心領域ごとの実装位置を特定できる (Priority: P2)

開発者は音声、表示、通信、OTA、ボード差分などの関心領域ごとに、どこに関連実装があるかを技術文書から特定できる。

**Why this priority**: 全体概要だけでは保守や改修の着手点がわからず、実際の作業に直結しないため。

**Independent Test**: 複数の関心領域を挙げて、技術文書だけで関連ディレクトリと代表的なファイル群を特定できることを確認する。

**Acceptance Scenarios**:

1. **Given** 開発者が音声や通信に関する修正を検討している状態で, **When** 技術文書を参照する, **Then** 各機能領域の関連モジュールと責務の境界を見つけられる。
2. **Given** 開発者がボード依存差分を確認したい状態で, **When** 技術文書を参照する, **Then** 共通実装とボード別実装の置き場所を区別できる。

---

### User Story 3 - 調査結果を継続的に再利用できる (Priority: P3)

チームは今回の解析結果をリポジトリ内の恒久的な文書として保持し、今後のオンボーディング、設計判断、調査の再出発点として利用できる。

**Why this priority**: 解析内容が口頭や一時メモに留まると再利用性が低く、同じ読解コストが繰り返し発生するため。

**Independent Test**: リポジトリを新規に取得したメンバーが、特別な補足説明なしに文書の場所を見つけて利用できることを確認する。

**Acceptance Scenarios**:

1. **Given** リポジトリ利用者が後から参照したい状態で, **When** リポジトリ内を確認する, **Then** 技術文書が見つけやすい場所に保存されている。
2. **Given** 調査内容を更新したい状態で, **When** 文書を開く, **Then** 構造化された見出しにより追記・修正すべき箇所を判断できる。

### Edge Cases

- 一部の実装領域が外部依存や生成物に強く依存していて詳細を断定できない場合でも、未確認範囲と確認済み範囲が区別されていること。
- ボードごとの差分が多く全件の詳細説明が難しい場合でも、共通構造と代表的な差分パターンは文書化されていること。
- 実装を読んだ結果、名称と実際の責務が直感と異なる箇所があっても、文書はディレクトリ名ではなく観察された責務に基づいて説明すること。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST `docs/` 配下に、`xiaozhi-esp32` の主要ディレクトリと中核モジュールの責務を整理した技術文書を追加する。
- **FR-002**: System MUST 技術文書に、実行開始点から主要機能へ至る高レベルな処理の流れを、代表的な主要フローとして含める。
- **FR-003**: System MUST 技術文書に、少なくとも音声、表示、通信、設定、OTA、ボード差分の各観点から関連実装の位置と役割を記載する。
- **FR-004**: System MUST 技術文書に、共通実装とボード固有実装の境界を読者が判別できる説明を含める。
- **FR-005**: Users MUST be able to 文書を読めば、関心のある変更対象に対応する調査起点となるファイル群またはディレクトリ群を特定できる。
- **FR-006**: System MUST 文書内で、確認済み事項、推定事項、対象外事項が混同しないように記述上の区別を設ける。
- **FR-007**: System MUST 作成する文書を、後続の開発者が追加調査を反映しやすい構成で保持する。
- **FR-008**: System MUST 文書の粒度を全モジュール網羅ではなく、全体構造の説明と起動・通信・音声などの代表的な主要フローの説明に限定する。

### Key Entities *(include if feature involves data)*

- **構造解析ドキュメント**: `xiaozhi-esp32` の全体構造、責務分担、主要フロー、関心領域別の実装位置を記録する恒久文書。
- **関心領域**: 音声、表示、通信、設定、OTA、ボード差分など、読者が目的別に参照したい技術的な切り口。
- **実装要素**: ディレクトリ、モジュール、代表ファイル、設定入口など、構造解析の対象となるコードベース上の単位。

### Allowed Scope

- `xiaozhi-esp32` の構造把握に必要な調査と、その結果をまとめた技術文書の新規追加または既存文書への追記。

- 実装の正しさ検証や性能評価を目的としたテスト整備。
